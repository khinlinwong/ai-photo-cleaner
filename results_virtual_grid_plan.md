# AI Photo Cleaner Results 虚拟网格与缩略图懒加载规划 - CORE-PERFORMANCE-2-PLANNING

## 一、 当前问题

在前几轮 JPG / PNG / WebP 混合格式真实图片的 true 分支物理压测中：
- 100 张、200 张与 300 张测试均已跑通并完成功能闭环，校验 Parity 数据 100% 一致。
- 但在 300 张物理压测下，`/results` 结果页网格大列表卡片由于一次性生成过多的 DOM 节点，首次跳转加载时主线程占用极高，且在滚动网格时出现了体感可见的轻微滚动掉帧。

### 核心研判
1. **渲染层为当前瓶颈**：当前相似检测算法的逻辑正确度已完成小、中批量测试，证明数据转换、适配与 Photo Battle 擂台状态流转符合架构预期。目前的系统瓶颈主要是结果页的大列表 DOM 膨胀与海量缩略图同时加载引发的主线程负载过重。
2. **测试截断**：在解决主线程渲染性能与大网格假死隐患前，不应继续进行 500+ 图片的盲测。
3. **低风险演进优先**：为了不引入高开销的架构更改，第一步应优先解决 results 结果页的可见区域（Viewport）渲染与缩略图懒加载，而不是优先做后台 Web Worker 多线程计算。

---

## 二、 优化目标

本规划旨在设计一个零第三方库依赖、低风险的 UI 渲染层局部优化方案，并严格坚守以下物理隔离边界：

1. **虚拟渲染**：`/results` 页面仅实例化渲染当前视口可见区域及其附近的卡片组件。
2. **节点瘦身**：极大降低页面整体 DOM 树的节点数，使 300+ 张照片在 results 下滚动的 DOM 开销稳定于常数级。
3. **内存削峰**：控制图片同时加载的数量，避免数百张高清大图瞬间解码抢占浏览器主存。
4. **不修改核心算法**：绝对不触碰已有的清晰度/曝光度评分、dHash 感知聚类等分析层算法。
5. **不改动状态机**：绝对不改动 Context 内的 Photo Battle 擂台对决控制队列、用户显式决断 `userDecision` 记录等逻辑。
6. **不改变 ZIP 逻辑**：ZIP 安全导出分区和 Results 网格划分规则继续复用 `getUserVisibleBucket`，保持完全同步。
7. **二值分类收敛**：用户的最终决断分类依然强制收敛为“保留”与“淘汰候选”，不存在任何技术性中间状态暴露。
8. **交互原样保留**：用户可见的交互操作（标记保留/淘汰、跳转擂台、查看详情）行为体验 100% 不发生变化。

## 三、 虚拟网格方案（Virtual Grid）

为了避免引入第三方依赖增加打包体积和状态传递的嵌套深度，建议采用**自研轻量级 CSS Grid + JS 虚拟网格方案**。

### 1. 核心设计思路
- **等高卡片网格**：结果页卡片视觉采用等宽等高设计，高度在布局上固定（例如设定固定高度 `320px`，含图片区域及操作面板）。
- **列数动态计算**：根据结果页容器宽度 `containerWidth` 和卡片最小宽度 `cardMinWidth`（如 `200px`），动态计算当前列数：
  $$\text{cols} = \lfloor \text{containerWidth} / \text{cardMinWidth} \rfloor$$
- **滚动边界计算**：通过监听容器的 `scroll` 事件，获取当前 `scrollTop` 和容器视口高度 `clientHeight`。
- **行索引计算**：
  - 起始行索引：$$\text{startRow} = \max\left(0, \left\lfloor \frac{\text{scrollTop}}{\text{rowHeight}} \right\rfloor - \text{overscanRows}\right)$$
  - 结束行索引：$$\text{endRow} = \min\left(\text{totalRows}, \left\lfloor \frac{\text{scrollTop} + \text{clientHeight}}{\text{rowHeight}} \right\rfloor + \text{overscanRows}\right)$$
- **安全 Buffer（Overscan）**：设置 `overscanRows = 2` 到 `4`，预先渲染视口上方和下方 2-4 行卡片，确保用户快速滚动时不会出现白屏。
- **支撑填充（Padding Offset）**：使用顶部的空白占位高度 `topPadding = startRow * rowHeight` 和底部的空白占位高度 `bottomPadding = (totalRows - endRow) * rowHeight` 来撑开滚动条，确保滚动条的物理长度和原生渲染一致。

### 2. 双分区应用
虚拟网格应当各自独立作用于：
- “保留区”照片网格
- “淘汰候选区”照片网格

两个分区拥有独立的滚动容器及独立的高度测算。

## 四、 缩略图懒加载方案

在虚拟化 DOM 结构的基础上，为缩略图引入**按需生成与销毁机制**。

1. **可见区装载**：当且仅当卡片行进入虚拟网格的 `[startRow, endRow]` 渲染区间时，卡片组件才正式被 DOM 渲染，此时才通过 `URL.createObjectURL(file)` 创建用于内存引用的预览图链接，或者执行 `<img>` 链接加载。
2. **离屏销毁与 Revoke（第一版暂不实施）**：
   - **注意**：第一版设计中，暂不实施复杂的离屏 objectURL 懒销毁与重新生成。频繁的销毁和重建会导致大图片在滚动进出视口时不断触发浏览器 CPU 重新解码图片，不仅耗费资源，还会引起明显的卡片内容闪烁。因此，第一版仅专注于 DOM 的虚拟滚动渲染，objectURL 的回收仅在整个工作空间被卸载（如重新导入、返回桌面）时集中进行。
3. **擂台加载优化**：Photo Battle 擂台属于高亮弹窗，本阶段无需对擂台本身虚拟化，但擂台所调用的两张对局照片应与虚拟网格共享预览大图引用，不重复生成 ObjectURL。

## 五、 分区适配边界说明

results 页面当前由多个区域拼合而成，虚拟网格**只作用于照片卡片展示层**：

- **受用区域**：
  - “保留区”的卡片照片网格列表。
  - “淘汰候选区”的卡片照片网格列表。
- **不受用（常规渲染）区域**：
  - 顶部的统计及操作工具栏（DesktopToolbar）。
  - Photo Battle 对决擂台模态弹窗（内部同时仅存在两张图片，无性能压力）。
  - 安全 ZIP 导出面板与 Dialog。
  - 技术详情 JSON 预览浮层。

## 六、 用户交互保持不变

虚拟化网格组件仅在 DOM 节点层面进行渲染层过滤与撑高占位，以下全部业务链路必须保持完全无感与不受影响：

1. **状态更新**：点击卡片下方的“移入淘汰候选”或“保留”按钮时，照片的 `userDecision` 属性被更新，Context 重新执行 `getUserVisibleBucket` 划分。网格数据源改变，虚拟网格会自动以新的列表重新计算滚动高度和可见行。
2. **擂台启动**：进入 `/results` 时，如果 `needsBattle` 为 `true`，Photo Battle 对决擂台的自动弹出不受虚拟网格的存在影响，依然基于 Context 中的 `similarGroups` 状态正常唤起。
3. **技术详情折叠**：点击卡片的技术详情折叠面板后，展示的信息数据来自 React 的 `photos` 状态数组，而不是从 DOM 节点抓取，保证数据绝对准确。
4. **ZIP 安全导出**：点击 ZIP 导出时，JSZip 计算的依然是 Context 层面的物理状态，与当前 DOM 树上是否虚拟化了节点完全解耦，导出内容依然 100% 对齐 results 分区。

## 七、 潜在风险点与规避方式

自研虚拟网格在实现上面临以下 4 大风险，必须在前置设计中做好规避：

1. **卡片动态高度抖动**：
   - *风险*：如果卡片的中文 Badges（如“相似照片”、“模糊风险”）数量不同导致卡片高度动态伸缩，会使高度测算崩溃，造成滚动闪烁和跳跃。
   - *规规避*：强制设定卡片高度为**固定绝对物理高度**。图片容器设置固定比例，文字及 Badge 区域设为溢出隐藏或固定高度，使 `rowHeight` 保持为硬性常量。
2. **快速滚动白屏（Flash of Unrendered Content）**：
   - *风险*：用户极速拖动滚动条时，JS 事件回调被延迟，可能导致视口局部白屏。
   - *规避*：适当增大 `overscanRows` 的缓存行数（如设为 4 行），并且在组件外部包裹一层极简的占位骨架屏（Skeleton Card）。
3. **切换分区后的可见范围失效**：
   - *风险*：从保留区切到淘汰候选区时，若不刷新滚动位置，可能导致新列表的行数越界，页面白屏。
   - *规避*：在分区视图切换、或数据源长度发生突变时，显式将滚动容器的 `scrollTop` 归零或重新触发 `triggerScrollUpdate` 以对齐重置。
4. **ObjectURL 过早释放**：
   - *风险*：快速滑动卡片进出视口时，如果频繁地 revoke 和 create，会导致浏览器 CPU 解码和内存 I/O 被撑爆。
   - *规避*：ObjectURL 不在单个卡片移出时瞬间释放，而是采取**轻量引用计数**或在整个 Desktop/Workspace 生命周期内统筹管理，仅在卡片确被销毁、重新上传或卡片量超过 300 张时在后台分批释放。

## 八、 验收通过标准

在未来实施 results 虚拟网格后，必须通过以下 10 项严苛指标的回归验收方为通过：

1. 100 张、200 张结果页加载完全正常，照片卡片分类与以前完全等价。
2. 300 张混合格式真实大图载入后，`/results` 页面的滚动性能显著提升，没有感知上的重绘卡顿与掉帧。
3. results 首次加载时 DOM 树节点个数大幅降低（对于 300 张图，可见区 DOM 稳定在 30 个以内）。
4. 卡片下方的操作按钮（保留/淘汰）触发状态映射正确无误，移动分区快速灵敏。
5. Photo Battle 擂台自动唤醒触发无阻，对决表决后卡片分类完美迁移。
6. ZIP 导出包的文件分区一致性保持 100%，没有产生未决照片的误漏。
7. 照片文件后缀和 JPG / PNG / WebP 原格式 100% 保留。
8. 整个优化只改动 results 组件或 results 相关的轻量 UI，**绝对不修改核心分析算法及 Context**。
9. 开关 `USE_SIGNAL_GROUPS_FOR_BATTLE` 回归校验保持 `false`。
10. `npm run build` 和 `npm run lint` 成功。
