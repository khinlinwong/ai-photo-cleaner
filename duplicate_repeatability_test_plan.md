# AI Photo Cleaner 重复运行稳定性测试规划 - CORE-DUPLICATE-REPEATABILITY-PLANNING

## 一、 测试目标

本阶段测试的核心目标**不是继续扩大单次测试的照片体量和压力**，而是验证在本地开发环境下，同一批测试集在 `USE_SIGNAL_GROUPS_FOR_BATTLE = true` 分支下**连续重复运行 3-5 次**时的整体性能和逻辑稳定性。

通过多轮循环导入和导出操作，重点验证以下十个维度：
1. **Parity 一致性**：每轮提取的算法比对 Parity 数据是否绝对保持一致。
2. **相似组数稳定性**：`oldSimilarGroupCount` 与 `newSimilarGroupCount` 在每轮中是否高度匹配且稳定。
3. **相似照片数稳定性**：`oldSimilarGroupedPhotoCount` 与 `newSimilarGroupedPhotoCount` 是否稳定对齐。
4. **组长不一致度**：`leaderMismatchCount` 必须在每一轮中都始终为 `0`。
5. **擂台流转流畅**：Photo Battle 弹窗和键盘/鼠标操作在每轮反复 PK 中是否流畅无死锁。
6. **分批打包稳定**：ZIP 分批导出是否在反复点击下均能稳定触发并正常串行排队下载。
7. **虚拟网格滚动**：`VirtualPhotoGrid` 网格在每一轮重入 results 页面后，其滚动加载与懒加载是否始终响应灵敏。
8. **内存防溢出**：利用 Chrome 任务管理器监控物理内存，确认是否出现每轮累加、持续上涨的失控趋势。
9. **句柄释放安全性**：确认在反复运行中，Canvas 图像数据、Blob 临时 URL、ObjectURL 以及下载链接句柄是否能在延迟 120 秒后得到正常、干净的销毁，无释放异常。
10. **长期运行稳定性**：验证在进入第二轮或第三轮时，系统是否会因为内存积压、垃圾回收滞后而出现明显的运行变慢、界面卡死、下载丢失等风险。

### 测试集 A：100 张大尺寸 JPG 压力集
- **存放目录**：`D:\ai-photo-cleaner-large-jpg-test` (位于项目外部)

- **验证重点**：用于在中等物理压力下，验证分批打包与 Object URL 延迟回收在多次重复重入时的可靠性。

### 测试集 B：200 张大尺寸 JPG 压力集
- **存放目录**：`D:\ai-photo-cleaner-large-jpg-test` (同上)
- **执行条件**：**仅当测试集 A 连续 3 轮重复性测试完全通过后，才允许启动此测试集验证。**
- **验证重点**：在较高物理压强下，确认多 part 分批 ZIP 串行排队下载机制（淘汰区 3 包，保留区 2 包）在反复调用时，是否在当前测试环境下依然能成功规避 OOM 与中断，而不是为了继续扩大规模。

### 可选测试集 C：mock 真实相册感 300 张数据集
- **存放目录**：`D:\ai-photo-cleaner-realistic-album-test` (位于项目外部)
- **数据特征**：包含大量连拍、轻微模糊、不同比例和曝光程度的模拟相册图片。
- **验证重点**：用于进一步验证复杂聚类拓扑在多次重入时的稳定性和 Parity 表现，作为后续阶段的候选测试。

## 三、 规划重复次数与执行步骤

本轮规划不执行测试，未来执行时需遵循以下梯度计划：

### 第一阶段：100 张大尺寸 JPG 连续运行 3 次
1. **测试前确认**：`USE_SIGNAL_GROUPS_FOR_BATTLE = false`，且工作区干净。
2. **开启灰度**：修改 `featureFlags.ts` 中常量为 `true`。启动 `npm run dev` 并打开 `http://localhost:3000/desktop`。
3. **第一轮操作**：
   - 选择并导入测试集 A 的 100 张大图，等待 `/processing` 扫描。
   - 在 `/results` 中记录 window QA parity 属性和内存基准。
   - 完成几组 Photo Battle 裁决，点击导出保留区与淘汰候选区 ZIP。
   - 确认下载完成（淘汰候选区自动拆为 2 包，保留区 1 包）。
   - 点击“重新导入”返回 `/desktop`，或手动刷新页面（Reset）。
4. **第二轮操作**：重复上述第一轮相同的导入、PK 与打包导出全生命周期动作，并记录指标。
5. **第三轮操作**：再次重复执行，完成后记录数据。

### 第二阶段：200 张大尺寸 JPG 连续运行 3 次
- 只有第一阶段 100 张测试 3 轮无任何报错、无内存泄露、Parity 100% 对齐时，才可开启此阶段。
- 采用测试集 B 重复进行 3 次“导入-处理-PK-分批 ZIP 导出（淘汰区 3 包，保留区 2 包）”的闭环操作。
- 只要其中任意一轮的任何一个步骤出现卡死、闪退、或者分包下载中断，必须立刻中止测试。

## 四、 规划每轮记录指标

在每轮测试的各个关键节点，必须详尽记录以下指标：

### 1. 基础流程指标
- `/desktop` 导入是否报错或卡顿。
- `/processing` 页面运行是否正常。
- `/processing` 扫描总耗时。
- `/results` 页面加载及 DOM 树渲染是否正常。
- results 页面虚拟网格快速拖拽滚动是否流畅、有无白屏。
- 浏览器是否发生无响应、崩溃或白屏。
- 控制台是否有非预期的 Error 日志。

### 2. QA Parity 对齐指标
- 验证全局属性 `window.__AI_PHOTO_CLEANER_QA__` 是否存在。
- 记录新旧算法组数及不匹配状态：`oldSimilarGroupCount`、`newSimilarGroupCount`、`similarGroupCountMismatch`。
- 记录新旧算法相似图片数及不匹配状态：`oldSimilarGroupedPhotoCount`、`newSimilarGroupedPhotoCount`、`similarGroupedPhotoCountMismatch`。
- 记录组长错配数 `leaderMismatchCount`。
- 校验 `generatedAt` 毫秒级时间戳是否在每轮生成时正常更新更新。
- 校验 `source` 属性是否为 `duplicateGroupQA`，且未包含任何文件物理绝对路径或 Base64 等泄露字段。

### 3. Photo Battle 交互指标
- 比对擂台是否自动弹出，界面有无错位。
- 物理点击“保留左边 (keep_left)”、“保留右边 (keep_right)”、“两张都保留 (keep_both)”、“两张都淘汰 (cull_both)”、“跳过 (skip)”是否全部流转无卡死。
- reset 擂台重置是否正常，数据状态有无发生漂移或紊乱。

### 4. ZIP 导出指标
- 保留区 ZIP 导出是否成功，记录生成的 part 数量和物理文件名。
- 淘汰候选区 ZIP 导出是否成功，记录生成的 part 数量和物理文件名。
- 每个分批 part 压缩包是否全部成功触发并完整下载至本地。
- 是否在下载途中出现 `DownloadInterrupted` 错误。
- 解压后所有 part 的合计照片数是否与 UI 分区展示的总张数绝对一致。
- 确认解压出的所有图片均保留了原始的 `.jpg` 后缀且文件完整。

### 5. 资源消耗观察指标（重点）
- **每轮开始前内存**：导入前，开发页面初始状态下的内存占用（使用 Chrome 任务管理器观察），作为每轮的基准内存状态。
- **扫描/打包/导出峰值内存**：
  - `/processing` 阶段 Canvas 诊断分析时的内存峰值。
  - JSZip 在主线程执行分批 `generateAsync` 压缩并生成 Blob ZIP 导出时的物理堆最高内存占用。
- **回收后内存（需等待延时释放）**：分批打包及下载流程彻底结束并**等待至少 120 秒后**再行记录。
  - *原因*：当前分批 ZIP ObjectURL 采用 120 秒延迟释放策略，如果不等待 120 秒，内存中可能仍包含尚未被 `revokeObjectURL` 释放的 Blob 数据，无法准确判断是否存在物理泄漏。
  - *监控方式*：对比每轮等待 120 秒之后的物理内存占用是否能正常回落到该轮“每轮开始前内存”附近。
- **判断内存泄露标准**：在每轮测试中建立“每轮开始前内存”、“ZIP 导出峰值内存”以及“每轮结束等待至少 120 秒后的回收后内存”三段监测记录。在分析内存泄漏时，应重点对比观察多轮的“回收后内存”是否呈持续的、无法回落的台阶式上涨。

4. **资源安全性**：每轮结束后内存能正常回收，无无法回落的持续性内存上涨。
5. **配置与隔离**：测试结束后 feature flag 开关常量必须顺利且物理恢复为 `false`，无 true 配置污染。
6. **Git 清洁度**：无任何测试图片、临时 ZIP 或是编译日志文件被提交到 Git，工作区保持 clean。

### 200 张重复性通过标准
1. **前置条件**：仅在 100 张大尺寸测试 3 轮顺利通过后执行。
2. **测试完成度**：连续 3 轮流程全部跑通。
3. **分批有效性**：每轮均能自动拆分为正确的多个分包，全部 part 成功下载，无 `DownloadInterrupted` 中断。

5. **内存无崩溃**：内存占用没有出现随轮次呈倍数递增或导致浏览器崩溃的危险态势。

## 六、 规划中止条件与失败处理

### 1. 中止条件
在多轮循环测试中，一旦遭遇以下任一情况，**必须立刻手动中止测试**：
- 出现任何 Parity 不对齐（group 数量或照片数量不匹配）。
- 出现组长 ID 错配（`leaderMismatchCount > 0`）。
- Photo Battle 出现状态死锁，或卡片无法正常向 zones 分区划归。
- 导出 ZIP 出现 `DownloadInterrupted` 错误，或分批 part 压缩包下载丢失。
- 多个 ZIP 解压后的文件总数与 Results 展示的数据不相符。
- 浏览器出现页面白屏、假死、内存溢出崩溃。
- Chrome 任务管理器显示内存持续上涨且页面重置后完全无法回落。
- 遇到不可逆的底层报错。
- 开关常量无法正常恢复或误将测试图片、临时 zip 暂存进了 Git。
- results 页面中出现了“保留”与“淘汰候选”以外的第三种最终导出状态。

### 2. 失败后处理规程
- **严禁擅自修改业务代码**：发现错误后，禁止在没有理清原因前盲目改动 Context、results 或是 ZIP 部分的业务源码。
- **一键物理回退**：立刻将 `USE_SIGNAL_GROUPS_FOR_BATTLE` 修改回默认 `false`，保证开发环境秒级无感回退。
- **详实数据记录**：记录失败的测试集、故障轮次、具体的 QA 指标偏差、错误发生的步骤以及控制台完整的 Call Stack 报错日志，并抓取 Chrome 内存占用峰值。
- **提报只读审查**：将收集的数据提交给 Codex，进入 `FAIL-READ` 阶段进行只读性安全审查和代码逻辑缺陷分析。

## 七、 规划 Feature Flag 规则

1. **初始状态拦截**：在测试执行前，必须确认配置文件 `src/lib/config/featureFlags.ts` 中 `USE_SIGNAL_GROUPS_FOR_BATTLE` 值为默认的 `false`。
2. **测试期临时集成**：仅在测试执行期间，允许在开发分支上手动将其修改为 `true`。
3. **整组复位原则**：在每轮测试结束切换间隙，不要求强制复位为 `false`；但在整组测试用例全部执行结束，或者测试中途发生故障中止时，**必须第一时间物理恢复为 `false`**。
4. **提交状态硬性约束**：最终运行 `git diff -- src/lib/config/featureFlags.ts` 必须输出为空。生产环境（production）绝不启用该灰度分支，锁定 legacy 稳定分流。

## 八、 规划 Git 安全规则与清洁度

1. **测试资源外置**：测试图片文件夹必须放在项目源码根目录外部，例如 `D:\ai-photo-cleaner-large-jpg-test`，绝对不允许复制或暂存进项目子目录中。
2. **下载目录重定向**：测试下载生成的 `keep_photos.zip`、`cull_photos_part_N.zip` 等压缩文件，必须下载在浏览器系统默认下载夹或项目外部，绝不进入项目子文件夹。
3. **防污染审查**：测试结束后，必须物理删除或移出所有下载的 zip 测试包，并运行 `git status --short` 进行最后清洁度校验。
4. **限制提交范围**：Git 的 commit 提交范围严格且精确限制在已规划允许修改的 markdown 文档文件上，不允许带入任何临时代码、临时 ZIP 或是测试图片的历史记录。
