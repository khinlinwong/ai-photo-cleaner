# AI Photo Cleaner Signal Groups Development-only 常态灰度评估 - CORE-DUPLICATE-SIGNAL-BETA-PLANNING

## 一、 当前已完成验证

在当前阶段，我们已通过多轮测试验证了 signal groups 分支的可靠性，具体包括：
1. **Demo true 分支测试通过**：Demo 旅行照片集（含相似组）在 `true` 分支下在当前测试中跑通，数据流和状态流转正常。
2. **35 张非隐私本地图片测试通过**：开发环境下小批量真实物理照片测试，没有发生逻辑偏差。
3. **200 张元数据仿真测试通过**：验证了较大量的照片流聚类和新旧算法对齐指标。
4. **100 / 200 / 300 张 BMP 文件测试通过**：验证了物理文件分析的算法正确性与基础读写逻辑。
5. **100 / 200 / 300 张 JPG / PNG / WebP 混合格式测试通过**：验证了多格式解码聚类表现。
6. **100 / 300 张 mock 真实相册感样本测试通过**：在更贴近日常手机相册（含连拍、模糊、过曝欠曝、不同纵横比等相册感特征）的 100 张和 300 张非隐私样本测试中，新旧算法的分组和照片指标在当前验证中已对齐。
7. **window.__AI_PHOTO_CLEANER_QA__ 可稳定读取 parity**：测试脚本通过全局 QA 属性能够稳定、无延迟地提取比对摘要。
8. **production 下 window.__AI_PHOTO_CLEANER_QA__ 不暴露**：生产包防污染拦截守卫生效，生产环境未暴露出任何 QA 属性。
9. **Photo Battle、ZIP、二值分类在当前测试样本下正常**：自动点击数轮对决，没有发生界面死锁或逻辑崩溃；ZIP 导出二值分区与页面划归完全一致。
10. **VirtualPhotoGrid 已改善 results 滚动表现**：自研虚拟网格大幅减少了 results 页面的 DOM 树节点个数和内存驻留，300 张测试下滚动流程流畅无异常。

## 二、 明确本阶段评估范围

本阶段**只评估是否可以进入 development-only 常态灰度**。

**development-only 常态灰度的明确定义：**
- **仅限开发环境开启**：仅在本地开发和测试环境允许更频繁地开启 `true` 进行测试或开发验证。
- **production 仍强制 legacy**：生产环境不直接默认开启此灰度开关。
- **不对普通用户启用**：不对任何线上最终用户暴露或开启。
- **不移除 legacy**：legacy 的 `detectDuplicates` 和 `similarGroups` 主流程代码继续保留，作为强安全备份。
- **不改变正式打包行为**：生产构建中自动闭锁，物理拦截阻断。
- **不改变用户最终分类**：用户决定的分类在任何时候依然只收敛为“保留”与“淘汰候选”的二值分类，无中间态污染。
- **灾备回退能力**：若开发环境出现任何异常，可一键将 `featureFlags.ts` 中常量改回 `false` 实现秒级无感回退。

## 三、 评估是否可以进入 development-only 常态灰度

### 建议结论
**可以考虑进入更严格的 development-only 灰度规划，但不建议立即把 `USE_SIGNAL_GROUPS_FOR_BATTLE` 默认改为 true。**

### 核心理由：
1. **多轮测试通过**：目前在开发环境通过了多轮小、中批量及真实相册感样本回归测试，两路算法在此样本下表现一致。
2. **校验手段健全**：`window.__AI_PHOTO_CLEANER_QA__` 提供了稳定的 Parity 自动校验闭环。
3. **缺乏真实客户相册长期测试**：当前的 mock 样本仅能模拟部分特征，仍未经历真实用户在各种操作系统、不同硬件环境下，海量复杂相册行为的长期、多样性实测。
4. **缺乏高分辨率手机原图压力**：目前的 mock 样本单张体积在 10KB~20KB，而手机真实照片（原图）单张可达数 MB 至十几 MB，在处理 300 张以上高分辨率手机原图时，Canvas 诊断与内存占用的物理压强尚未得到实测。
5. **未覆盖 HEIC / RAW 格式**：移动端极其常见的苹果 HEIC 文件和相机 RAW 格式在测试中依旧被隔离排除。
6. **production 稳定性优先**：在没有充分把握前，生产环境必须确保完全不出故障，不应盲目推进生产默认值。
7. **legacy 仍是必要回退路径**：传统主流程稳定可靠，随时准备在灰度逻辑发生故障时提供秒级切换兜底。

---

## 四、 规划可接受的下一步

建议下一步并不是直接将 `USE_SIGNAL_GROUPS_FOR_BATTLE` 默认值改成 `true`，而是评估以下三种技术方案：

* **方案 A：继续手动临时 true（推荐）**
  - **风险最低**：完全没有代码副作用。
  - **规程**：只在开发期回归测试前手动改写 `true`，测试完毕后立即恢复 `false`。
  - **缺点**：测试和开发需要手动改动常量，操作成本稍高。
* **方案 B：新增 development-only localStorage 开关规划（暂不采纳）**
  - **机制**：在 development 环境下读取 localStorage 中的配置值（例如 `localStorage.setItem('aiPhotoCleaner.useSignalGroupsForBattle', 'true')`）。
  - **当前结论**：为了保持代码绝对纯净、防范越权依赖，本阶段暂不实现 localStorage 逻辑。
* **方案 C：新增 dev-only UI debug switch 规划（不采纳）**
  - **机制**：仅在开发模式下的页面中渲染一个可视化切换 legacy/signal 的 debug 按钮。
  - **当前结论**：侵入 UI 组件树，极易引入展示层的代码杂质，风险偏高，不予采纳。

**决策建议与结论：**
本阶段**仅继续保持方案 A**（手动临时修改并即时恢复），**暂不实现任何新开关代码（包括不实现 localStorage 开关和 UI 调试开关）**。

---

## 五、 明确不做事项

本阶段及下一阶段，我们将恪守以下红线约束，不进行以下事项：
1. **不把 USE_SIGNAL_GROUPS_FOR_BATTLE 默认改 true**：配置文件中常量默认值继续锁定为 `false`。
2. **不启用 production true**：生产包和线上生产环境依然运行 legacy 主流程。
3. **不移除 legacy 逻辑**：不移除 `detectDuplicates` 及 `similarGroups` 链路。
4. **不把 signal groups 直接接管用户流程**：不使用无保护的新分支接管真正的生产用户流程。
5. **不添加用户可见开关**：任何灰度配置和 debug 选项绝不对最终用户暴露。
6. **不覆盖 HEIC / RAW**：暂时维持格式隔离，直至有完备的格式转换和解析规划。
7. **不进行 500+ 盲测**：避免在未进行性能架构优化前盲目扩大测试张数。
8. **不把 mock 测试结论扩大到真实客户相册**：始终承认人工模拟样本与真实复杂相册在体积、重复形式等方面的物理差距。
9. **不进入用户 beta**：由于缺乏更多格式和客户环境的验证，当前不建议面向用户进行任何形式的 beta 开启。

最终用户所见的照片分类，任何时候依然强制收敛为“**保留**”与“**淘汰候选**”两类。

## 六、 规划下一阶段更有价值的测试

在考虑任何常态灰度前，优先补充并落实以下三类测试规划：

1. **大尺寸手机原图测试规划**：
   - 使用 100 / 200 张单张大小为 3MB-10MB 的真实 JPG 照片。
   - 重点验证物理大图在主线程 I/O 读取、Canvas 像素解码、缩略图内存驻留以及 ZIP 多线程打包时的性能压强。
2. **更真实来源的非隐私公开图片测试规划**：
   - 绝不使用客户照片或私人隐私照片。
   - 采用开源公版（Unsplash/Pixabay）的风景、建筑、日常静物等真实细节大图，提高图像细节的熵值。
3. **长时间重复测试规划**：
   - 针对同一个测试集在浏览器中重复执行 3-5 次“导入-处理-results-PK-导出”完整生命周期。
   - 验证 Parity 摘要数字是否保持一致。
   - 监控浏览器内存占用情况，判断是否存在 Canvas 句柄或 objectURL 的内存泄露。

## 七、 规划通过标准

如果在未来阶段需要将 true 常态化（或开发环境默认 true），必须至少满足以下条件：
1. 真实相册感 100 / 300 张回归测试持续通过。
2. 大尺寸手机原图 JPG 100 / 200 张压力测试通过。
3. parity 指标在重复性测试中连续多轮表现一致。
4. Photo Battle 状态机流转正常，无对局卡死或 Dialog 遮挡超时。
5. ZIP 安全导出分类正确，无空文件或解压失败。
6. results 分类保持二值分类，无极值以外分类。
7. production 生产构建中不暴露全局 QA 摘要。
8. production 依然强制锁定于 legacy 稳定路径。
9. 一键物理改写常量为 `false` 可在开发环境瞬间恢复。
10. 项目 Git 工作区干净，没有测试图片或 zip 被误提交。
