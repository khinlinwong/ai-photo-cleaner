当前开发环境的 Results 结果页面已完成了以下稳定化建设：
1. **VirtualPhotoGrid 接入**：成功引入了虚拟滚动，在处理大量照片时有效改善了 results 页面的 DOM 树节点个数和内存驻留，列表拖拽极为流畅。
2. **二值分类归纳稳定**：最终导出的分类关系牢牢收缩于“保留 (keep)”与“淘汰候选 (cullCandidate)”二值，逻辑逻辑防线完备。
3. **Photo Battle 状态流转正常**：相似照片 A/B PK 能够自动弹出、键盘/鼠标操作及 Esc 安全退出均在当前验证中无卡死故障。
4. **分批 ZIP 导出已实现**：串行打包和 3000ms 下载延迟成功规避了浏览器 OOM 崩溃危险。
5. **ZIP 大包提示与失败 warning 已挂载**：常驻说明、大相册提醒、isZipping 进度提示及 catch 块友好错误引导全部成功在 [results/page.tsx](file:///C:/Users/khinl/Documents/AI%20Photo%20Cleaner/src/app/results/page.tsx) 中集成，100 张大尺寸 JPG 回归测试通过。
6. **200 张大图 ZIP 物理边界确立**：网页原型超大体积导出中断问题已确定属于浏览器 Blob 物理天花板限制，不再盲调参数。
7. **UI Polish 紧迫性**：当前下一步重点从算法对齐和物理压测，转移为结果页面的产品体验细节润色，统一文案口径、优化层级、设计空状态，让用户更清楚地理解相册状态和下一步操作。

## 二、 规划 UI Polish 目标

本阶段以纯视觉与展示层微调为主，目标设定如下：
1. **提高视觉层级可读性**：梳理各区域对比，使页面结构更加清晰。
2. **统一“保留 / 淘汰候选”口径**：全面排查并禁用有 SaaS 歧义的“删除”、“AI 废片判定”等表述，降低心理负担。
3. **降低用户对分批 ZIP 的误解**：改善多 ZIP 连续下载的文案连贯性，防止用户刷新或误认为出错。
4. **Photo Battle 入口清晰化**：使 A/B 对局入口更加显眼，并让用户明白对战只是为了辅助决策，不自动删除磁盘原图。
5. **按钮状态高可读性**：优化 isZipping、导出区为空等各种条件下的按钮 hover 及禁用视觉反馈。
6. **空状态 (Empty States) 完善**：补充保留区、淘汰候选区和相似组为空时的友好图案与文案提示。
7. **坚守低风险**：纯粹修改 Next.js page 页面的 CSS 样式与文案标签，**绝不触碰核心聚类算法、分区规则及 ZIP 导出底座。**

## 三、 规划检查与微调区域

### 1. 页面标题区
- **现状**：缺乏大局感引导说明。
- **Polish 规划**：在 results 页面顶部添加简短副标题，例如：
  > “请在此最终确认要保留和标记为淘汰候选的照片。本步骤仅在浏览器中整理，绝不物理修改您的原图文件。”
- **红线**：严禁在此区域出现任何“删除照片”或“清理磁盘”的误导性词汇。

### 2. 统计摘要区
- **现状**：保留照片、淘汰候选照片与相似组统计并列，对比不够突出。
- **Polish 规划**：优化统计卡片的外观和边框，使“保留照片数”与“淘汰候选照片数”具有明显的视觉对比（如绿色与橙红色角标）。如果仍有未比对完毕的相似组，在此区域追加强提醒，提示用户有未决定的相似大组。

### 3. ZIP 导出区
- **现状**：常驻提示、照片数量警告、isZipping 提示及 catch warning 在垂直排列时可能略显局促。
- **Polish 规划**：优化提示文本的纵向间距（`space-y-1.5`）与上边框分割线，利用小圆点或小灯泡 icon 区分：
  - 常驻说明（普通信息，灰色小字）
  - 大相册警示（警告信息，黄色小字，大于 100/200 张时展示）
  - 导出失败 warning（错误信息，黄色警告框，仅在 catch 块捕捉异常时弹出）
- **红线**：禁止修改 downloadPhotosZip 底层分包及 revoke 逻辑。

### 4. Photo Battle 入口
- **现状**：按钮文字“继续 PK 对比”可能让新手难以理解 A/B 擂台的本意。
- **Polish 规划**：优化文案并丰富说明信息，例如：
  - 按钮文案：“进入 A/B 对局对比” 或 “继续 A/B PK 对局”。
  - 辅助小字：“逐张比较相似照片，由您做出细致的保留或淘汰选择，不自动删除。”
- **红线**：禁止修改 Photo Workspace Context 中的对决状态机与键盘快捷键。

### 5. 保留区 / 淘汰候选区标题与分区
- **现状**：部分旧代码或注释残留 `delete` 口径。
- **Polish 规划**：全面排查 UI 组件树，确保所有对用户暴露的分区标题、卡片角标与表格表头均统一翻译为“保留区”与“淘汰候选区”。
- **英文口径对应**：
  - 保留：`Keep`
  - 淘汰候选：`Cull Candidate`
  - **红线**：禁用 `Delete` / `Trash` / `Junk` 产品口径，防止用户产生磁盘原片被删的恐慌。

### 6. 列表空状态 (Empty States)
- **现状**：若某个区照片数为 0，会直接展示空白，影响观感。
- **Polish 规划**：
  - **保留区为空**：展示“暂无保留照片。您可以通过 A/B 对局对比或在结果卡片上进行移动调整。”
  - **淘汰候选区为空**：展示“暂无淘汰候选。目前所有照片都处于保留区，安全系数 100%。”
  - **相似组全部 PK 完毕**：在 details 卡片中展示“相册中未发现明显的相似照片组。所有照片已完成归类。”

### 7. 导出按钮状态
- **现状**：`disabled` 状态样式稍显突兀。
- **Polish 规划**：优化按钮置灰（`opacity-40` 与 `cursor-not-allowed`），并且当按钮因 isZipping 或列表为空而 disabled 时，在按钮下方显示小字说明（如“正在导出中，请静候下载...”），防止用户重复点击。

### 8. 小屏与自适应布局
- **现状**：在 768px 以下的平板或窄视口上，侧边卡片与底部分包提示有少量间距挤压风险。
- **Polish 规划**：微调 margins 边距，对重复提示的文案进行提炼，避免在小屏下警告文字多行重叠撑坏 Flex 栅格。

## 六、 规划验收标准

未来实现阶段通过标准如下：
1. **标题口径统一**：页面主副标题清晰表达“保留照片”与“淘汰候选照片”，无“删除”等歧义。
2. **统计卡片错落有致**：统计摘要区颜色鲜明、指示性强。
3. **ZIP 提示层级分明**：常驻小字、大相册数量警告、isZipping 提示在纵向上不挤压，错误 warning 边框美观。
4. **PK 入口解释详细**：Photo Battle 按钮下配有清晰无公害的 A/B 比较机制说明。
5. **空状态支持**：保留照片、淘汰候选照片与相似组为空时，展现友好的缺省文案。
6. **禁用态自恰**：导出按钮禁用时，视觉反馈良好，无重复点击隐患。
7. **小屏响应式自适应**：在窄视口下无任何排版挤占或撑坏情况。
8. **逻辑与 Parity 零影响**：不污染 duplicate 双路对齐（错配数始终为 0）与 ZIP 分包文件名。
9. **USE_SIGNAL_GROUPS_FOR_BATTLE 保持 false**，不安装外部包。

## 七、 实现结果与总结 (CORE-RESULTS-UX-POLISH)

本规划已于本阶段成功在 [results/page.tsx](file:///C:/Users/khinl/Documents/AI%20Photo%20Cleaner/src/app/results/page.tsx) 中落地实现，核心成果总结如下：
1. **标题与副标题区**：在 results 页面顶部增加了大局标题“整理结果”与清晰的提示副标题，向用户明确说明仅为浏览器中整理，绝不物理修改原图文件。
2. **统计摘要区**：实现了由 `keepPhotos`、`deletePhotos`、`similarGroups` 与 A/B 对局完成状态组成的四列精致卡片统计栏，对保留/淘汰候选大类进行了突出的数值与体积呈现。
3. **ZIP 提示层级优化**：彻底重构了 ZIP 提示部分的 UI 排版。当 isZipping 时展示精美的呼吸动画条；catch 块捕获的失败 warning 用更加严密的黄色边框警告框展示；常驻与大相册警示合并为了简洁的 `black/10` 提示卡片，层级十分清晰。
4. **Photo Battle 入口及文案**：侧边卡片在有相似组未 PK 完时提供高亮按钮“继续 A/B 对局”并增加无害 A/B 机制的辅助解释小字，且在没有相似组时优雅地展示“当前没有需要 A/B 对比的相似组”；顶部的进度提示条同步将“继续 A/B 对比”统一为“继续 A/B 对局”。
5. **“保留 / 淘汰候选”口径统一**：将 Bucket 的名称从“保留区/淘汰候选区”统一翻译为“保留照片/淘汰候选照片”用于更精准的口径标定，且过滤及下载逻辑保持原状，未引入“删除”等误导性词汇。
6. **空状态适配**：完成了对保留区照片为 0、淘汰候选照片为 0 的自定义缺省说明文案挂载，当相似组为 0 时 PK 进度条显示为“未发现明显相似组...”。
7. **按钮 disabled 说明**：导出按钮在 `disabled`（因为为空或 `isZipping`）时在按钮正下方渲染对应的小字提醒（如“暂无可导出的保留照片”或“正在生成 ZIP，请等待当前导出完成。”），防止用户重复点击。
8. **小屏布局微调**：所有卡片和列表均采用 Flex 与百分比栅格宽度，提示信息采用 `leading-normal` 与 `text-[9.5px]` 紧凑规格，窄屏下自然换行，无任何挤压。

### 绝对红线约束核对：
- **未改动 Context**：PhotoWorkspaceContext 状态机、PK round 轮次、`updatePhotoStatus` 与 `applyBattleDecision` 接口等未做任何修改。
- **未改动 Photo Battle 状态机**：键盘 Esc、左右方向键等触发机制保持零变动。
- **未改动 analysis / duplicate**：未修改客观清晰度与聚类代码，未改变新旧相似算法 parity 指标。
- **未改动 getUserVisibleBucket 与 ZIP 导出分包**：分区算法、分包 build 逻辑、revoking 延时时间常量全部保持原封不动。
- **未安装新依赖**：未引入任何 UI 组件库或额外的 npm 包。
