# AI Photo Cleaner ZIP 导出架构升级规划 - CORE-ZIP-EXPORT-ARCHITECTURE-PLANNING

## 一、 失败背景记录

当前网页原型的 ZIP 导出机制已历经四个主要阶段的尝试：

### 阶段 1：单个大 Blob ZIP
- **机制与参数**：通过 JSZip 在主线程一次性生成单个完整的 ZIP 包 Blob。
- **200 张大尺寸 JPG 测试表现**：淘汰候选区生成 1GB+ 的超大单个 Blob，触发 `DownloadInterrupted` 导致下载失败。
- **根因分析**：同步打包巨型 Blob 内存开销极易击穿浏览器物理 Heap，且 `URL.revokeObjectURL` 立即释放导致浏览器在异步磁盘 I/O 写入期间句柄失效。

### 阶段 2：延迟 revokeObjectURL 机制
- **机制与参数**：引入 `setTimeout` 将 ObjectURL 的销毁统一延迟 120 秒执行。
- **200 张大尺寸 JPG 测试表现**：中等大包（<1GB）的下载稳定性有所改善，但面对 1GB+ 淘汰区大包仍不可靠，依然会频繁触发 `DownloadInterrupted`。
- **根因分析**：延迟释放能规避句柄同步抢跑，但无法治愈巨型 Blob 引发的内存峰值过载问题。

### 阶段 3：分批 ZIP 导出架构
- **机制与参数**：通过 `async/await` 串行完成分包打包，设置局部常量 `MAX_ZIP_BATCH_BYTES = 500MB`，`MAX_ZIP_BATCH_PHOTOS = 50`，`ZIP_BATCH_DOWNLOAD_DELAY_MS = 1500ms`。
- **测试表现**：100 张大尺寸 JPG 顺利通过；但在 200 张大尺寸 JPG 重复性稳定性测试第一轮（Round 1）打包导出 `cull_photos_part_3.zip` 时依然发生 `DownloadInterrupted`。
- **根因分析**：500MB 单包阈值偏高，多个分包在 120 秒延迟窗口内累积，JSZip 打包 Peak 内存最高冲到 `4454.17MB`。

### 阶段 4：参数调优测试
- **机制与参数**：收紧常量至极其保守的 `300MB` / `30 张` / `3000ms`，保持 120 秒延迟释放。
- **测试表现**：100 张大尺寸 JPG 重复性测试 3 轮完美通过；但 200 张大尺寸 JPG 第一轮导出第 4 个分包 `cull_photos_part_4.zip` 时，再次触发了 `DownloadInterrupted`。Peak 内存降低到了 `3931.92MB`，但未能根治中断。
- **结论**：分批 ZIP 架构能有效应对中等压力导出（如 100 张大图），但对于 200 张超大相册（体积超 1.27GB，连同保留区共 7 包），分批越细虽然单包更安全，但包数量激增，多 part 连续排队下载仍会使浏览器 I/O 管道超载。盲目在网页原型上微调参数已达到边际效应递减的死胡同。

## 二、 明确不是哪些问题

本次 200 张大图多 part ZIP 导出失败**不指向**以下系统功能：
1. `duplicate` / `analysis` 相似聚类与客观分析算法错误。
2. 双路比对 QA Parity 一致性校验错误（100 张与 200 张在所有测试轮次中 Parity 指标均 100% 保持完美对齐，`leaderMismatchCount = 0`）。
3. Photo Battle 擂台赛状态机与交互逻辑错误（对局流转与 Esc 安全关闭一切正常）。
4. `getUserVisibleBucket` 照片可见性分类分区规则错误。
5. 特征开关 `USE_SIGNAL_GROUPS_FOR_BATTLE` 已物理恢复为 `false`，生产环境继续保持 legacy 稳定分流，无代码污染。

## 三、 分析浏览器 ZIP 当前边界与瓶颈

当前网页原型在处理超大相册导出时触碰到了以下浏览器固有物理瓶颈：
1. **主线程内存压力**：JSZip 在主线程生成大 Blob 时，因内存垃圾回收（GC）无法即时进行，瞬时内存开销（Peak 内存最高冲近 4GB）极易诱发浏览器内核 OOM 崩溃。
2. **ObjectURL 驻留叠加**：120 秒的延迟释放导致在生成 5-7 个分包时，多个 300MB 的 Blob 内存会短期并存在浏览器内存中，造成内存压力乘积式放大。
3. **并发写入管道压力**：Chromium / QtWebEngine 在连续接收到多个排队大文件下载指令时，网络栈与磁盘 I/O 写入线程容易产生句柄冲突或写盘超载，抛出 `DownloadInterrupted`。
4. **下载状态不可控**：网页端 JS 无法可靠监听到每一个 ZIP 包是否在操作系统中“真正写入磁盘完成”，因而无法采取真正的“写完一包再打包下一包”的主动流控，盲目依靠 sleep 间隔（3000ms）在文件数量变多时容错率依旧很低。

## 四、 规划短期策略：保留分批 ZIP 并标注能力边界

1. **功能保留**：保留当前已实现的分批 ZIP 导出架构，将 `300MB / 30 张 / 3000ms` 参数作为开发环境下相对最保守的版本。
2. **能力边界标注**：承认证网页 Blob 下载在超大相册场景下的物理上限。网页版不将 200 张大尺寸 JPG ZIP 一次性通过作为必达目标。
3. **产品端提示升级**：
   - 在 Results 页面的导出 ZIP 区域增加文案提示，明确告知大相册将分批导出，且受制于浏览器性能限制。
   - 若导出发生中断，建议用户分多次导入或减少单次导出的选择张数。
   - 限制在产品文案中对超大单次导出（200 张以上大图）的稳定性承诺。

## 五、 规划中期方案 A：ZIP Web Worker 方案

### 目标与机制
1. 将 `JSZip.generateAsync` 压缩和打包流程从 UI 主线程剥离，移入独立的 Web Worker 线程。
2. 避免打包期间造成的 results 主页面卡死与 UI 白屏。
3. 支持打包过程中的进度更新（Progress）和用户取消（Cancel）功能。

### 优缺点评估
- **优点**：解放主线程，提高用户交互响应性，可复用现有的分批数据结构，改动量中等。
- **限制与代价**：Worker 线程打包大 Blob 依旧需要消耗相同的物理内存（只是移出了主线程），无法从根本上消除 Blob 在 120s 延迟释放窗口内的内存堆积风险，多 part 排队引发的 `DownloadInterrupted` 中断依然是浏览器下载管道控制的黑盒。

---

## 六、 规划中期方案 B：流式 ZIP / 分块写入方案

### 目标与机制
1. 避免一次性在内存中生成完整的巨大 Blob 压缩包，改为边压缩、边流式写入（Streaming）。
2. 利用现代浏览器的 `File System Access API`（支持 `showSaveFilePicker` 与 `FileSystemWritableFileStream`）由用户授权选择一个本地空闲目录或文件，直接分块流式写入本地磁盘。
3. 借助第三方流式打包类库（如 `zip-stream` 或 `StreamSaver` 架构）进行直接落盘。

### 优缺点评估
- **优点**：内存峰值会极低，不占用巨大的 Blob 运行栈，从根本上避开了 Chromium 内存溢出上限，非常适合超大批量图片的本地导出。
- **限制与代价**：`File System Access API` 在低版本浏览器及不同宿主 WebView 容器上的兼容性不稳定；流式写入对于本地系统权限和交互控制更为复杂，需要引入新的依赖。

---

## 七、 规划长期方案 C：Tauri / Native Export 原生文件操作方案

### 目标与机制
1. 在未来的 Tauri 桌面架构迁移中，彻底摒弃网页端的 Blob ZIP 下载机制。
2. 借助 Rust 后端原生文件系统操作能力，实现直接在用户本地磁盘上进行文件的复制与移动。
3. 提供原生 ZIP 压缩导出，或者直接创建本地 `保留相册` 与 `淘汰候选相册` 物理文件夹，将图片完美划归，彻底消灭浏览器 Blob 和下载管道的所有瓶颈。

### 优缺点评估
- **优点**：性能极强，文件复制时间短（秒级完成），最符合 AI Photo Cleaner 作为本地桌面软件的最终定位，物理隔离最彻底。
- **限制与代价**：属于整体框架重构，需要进入 Tauri 后端开发阶段才可实施，不是 Next.js 前端 MVP 的轻量改动。

## 八、 规划产品层面的导出策略

基于现阶段的技术约束，产品侧应做出以下调整以确保灰度与测试可以平稳过渡：
1. **Beta 准入条件与导出瓶颈界定**：
   - 网页端 200 张大尺寸 JPG ZIP 全量稳定导出不再作为当前浏览器原型的硬性完成目标，但在考虑 beta 之前，必须先完成清晰的 UX 限制提示、导出边界说明、失败引导和用户分批操作建议。在这些产品端的限制与引导未完成前，不放宽 beta 准入判断。
   - 将网页版 Beta 准入指标更新为：**100 张大尺寸 JPG 连续 3 轮重复性稳定性测试 3 轮完美通过，且 200 张大尺寸 JPG 在开发灰度分支下的客观分析、双路 parity 校验以及擂台赛 PK 操作全部通过，同时 results 页面上必须完成清晰的容量限制提示、失败容灾引导与分批导出操作提示。**
   - 200 张 ZIP 多 part 下载失败明确定义为浏览器原型的导出能力边界风险，而非阻碍核心算法演进的门槛。
