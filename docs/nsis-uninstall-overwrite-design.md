# NSIS 卸载与覆盖安装流程设计方案：CORE-DESKTOP-NATIVE-NSIS-UNINSTALL-OVERWRITE-DESIGN-1

## 1. 背景与问题描述

在 AI Photo Cleaner 桌面端（Native）打包测试中，用户在双击运行 NSIS 安装包（`AI Photo Cleaner_0.1.0_x64-setup.exe`）并尝试卸载或覆盖安装时，遇到了以下非预期行为：
- **核心现象**：执行卸载流程结束后，系统没有关闭安装器窗口并正常退出，而是直接又开始自动进入安装流程。
- **用户预期**：
  1. 用户通过 **Windows 设置 -> 应用 -> 安装的应用** 或 **控制面板** 卸载时，卸载完成后应彻底结束并关闭窗口，绝不能自动拉起安装程序。
  2. 用户双击运行同一个 `setup.exe` 覆盖安装或更新时，应明确提示是覆盖安装/修复/重新安装，而不是在执行完旧版本卸载后，在不经用户二次确认的情况下直接静默开始全新安装，甚至误触发卸载与安装的死循环。

本方案旨在通过只读审计 Tauri v2 的打包配置，分析导致此现象的深层原因，记录 NSIS 与 MSI 的手动复现测试矩阵结论，明确交叉安装的风险，并修正后续的修复与设计步骤。

## 2. 当前 Tauri / NSIS 配置诊断

通过对项目根目录及 `src-tauri` 下核心配置文件的审计，诊断如下：

1. **基本字段审计** (`src-tauri/tauri.conf.json`)：
   - `productName`: `"AI Photo Cleaner"` —— 包含空格的正式品牌名称。
   - `identifier`: `"com.aiphotocleaner.app"` —— 已从默认的 `com.tauri.dev` 修正为合规的唯一标识符。
   - `version`: `"0.1.0"` —— 语义化版本号。
2. **打包目标审计**：
   - `"targets": "all"` —— 编译时会同时在 `src-tauri/target/release/bundle/` 下生成 `.msi` (WiX) 和 `-setup.exe` (NSIS) 两种格式。
3. **NSIS 特定配置审计**：
   - 当前 `tauri.conf.json` 的 `"bundle"` 下**不存在** `"nsis"` 配置块。
   - 项目**完全使用 Tauri 默认的 NSIS 模板与内置逻辑**进行打包，未配置任何自定义模板或钩子。
   - 未配置 `installMode`，默认采用 `currentUser` 模式（安装至 `%LOCALAPPDATA%`，无需管理员权限）。

### NSIS 安装模式说明：
- `currentUser`：当前用户安装，默认安装到 `%LOCALAPPDATA%` 且不需要管理员权限。
- `perMachine`：系统级安装，安装到 `Program Files`，通常涉及 UAC 管理员权限提升。
- `both`：在安装时允许用户选择安装范围。
- **配置规范**：当前项目如果要先做最小风险修复，应优先评估 `currentUser`。具体配置字段及其有效性必须以当前项目实际使用的 Tauri v2 / cargo-packager schema 为准。

## 3. 问题根源与定位

### 3.1 问题集中在 NSIS `setup.exe` 维护流程
通过对安装与卸载流程的隔离验证，得出以下明确的定位结论：
- **产品业务代码完全正常**：该问题与 React/TypeScript/Rust 等应用业务逻辑无关，应用层没有执行任何进程自杀或目录删除。
- **`uninstall.exe` 单独运行完全正常**：当从 Windows Apps / 控制面板单独触发 `uninstall.exe` 卸载时，卸载程序会安全清除所有文件和注册表项，随后**正常退出并关闭窗口，不会重装**。
- **问题根源**：问题完全集中在 NSIS `setup.exe` 的 `PageReinstall` / `PageLeaveReinstall` 维护/重装流程中。当 `setup.exe` 启动并检测到系统已安装该应用时，若用户触发了卸载旧版本逻辑，主安装包 `setup.exe` 运行完 `uninstall.exe` 后没有调用 `Quit` 或 `Abort` 强制退出，而是继续往下流转，导致了“卸载后又自动安装”的非预期循环。

### 3.2 MSI 格式包有效性及局限性说明
上轮测试中弹出 Windows Installer 用法说明窗口，是由于 `msiexec` 调用的命令行参数和引号逃逸冲突导致，**不代表 MSI 包体损坏**。
- **MSI 包经补测完全有效**：正确安装命令为 `msiexec /i "完整MSI路径"`（或通过 `msiexec /i "路径" /qn` 进行静默安装）。
- **MSI 卸载流程完全正常**：卸载完成后会自动关闭且不会重新安装。
- **MSI 局限性**：MSI 包的写入逻辑硬编码为 `HKEY_LOCAL_MACHINE` (HKLM) 注册表系统键值，需要强制提升管理员权限。在非提权环境下运行时，会因为权限不足抛出 `1603 (Access Denied)` 错误而中断。因此，**MSI 暂不适合作为普通用户的 Alpha 主分发包**。

## 5. 交叉安装风险警告

> [!WARNING]
> **严禁混用 NSIS 和 MSI 安装包进行测试或分发！**
> 
> 由于 NSIS（在 `currentUser` 下）向 `HKEY_CURRENT_USER` (HKCU) 写入卸载键，而 MSI 向 `HKEY_LOCAL_MACHINE` (HKLM) 写入卸载键，但它们的实际安装物理目录均指向 `%LOCALAPPDATA%\AI Photo Cleaner`。
> 
> 如果交叉运行它们：
> 1. 系统中会在 Windows Apps 列表下生成**两个完全重名**的卸载入口。
> 2. 两者实质上在共享并覆盖同一个物理文件夹下的 `app.exe` 等核心文件。
> 3. 一旦卸载其中一个，物理目录会被彻底清空，导致另一个没有文件，使其关联的卸载快捷方式和注册表项残留，程序损坏。
> 
> **Alpha 测试与发布阶段，必须只选择并固化一条分发通道（首选 NSIS  currentUser）。**

## 6. 安全边界与约束

任何打包修复和配置优化均**必须遵守以下红线约束**：
- **不用 app 业务代码解决 installer 问题**：软件本身的运行时代码不得处理安装和卸载的具体文件系统清理工作，这必须完全由 Windows 安装服务或 NSIS 卸载程序隔离执行。
- **不新增 Tauri 权限**：不可为了安装/卸载而在 `capabilities/default.json` 或 `tauri.conf.json` 中扩大权限范围。
- **不开启危险 API**：严禁开启 `fs:allow-all`、`shell:allow-all` 等广域危险权限。
- **不编写危险删除脚本**：不可在 NSIS 模板或钩子中加入可能导致用户照片丢失的递归删除命令。
- **绝对不触碰用户照片目录**：软件的安装与卸载逻辑与用户照片原件目录应物理隔绝，只能清除 App 安装释放的资源。
- **不影响 copy-only / report schema**：此安装器问题的修复不得改动 Copy-Only 整理输出机制与导出的 `report.json` 脱敏配置。
- **不影响 Native ZIP guard**：本地运行时对 ZIP 导出组件的安全置灰拦截逻辑必须原样保持。
