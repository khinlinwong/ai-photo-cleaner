# Native 更大批量处理安全设计规划（100 张）：CORE-DESKTOP-NATIVE-BATCH-SCALE-100-DESIGN-1

## 1. 背景
AI Photo Cleaner 目前在桌面端 (Native) 已经平滑演进到了最大支持 50 张小批量照片处理的稳定版本。为应对更进一步的 100 张照片中批量整理，我们需要在继续保障内存安全、隐私防泄露、UI 响应度及进程健壮性等前提下，对内存限制、进度管理、取消机制、失败统计以及回滚策略进行详尽的前置安全设计，作为后续 100 张扩容的技术实施规范。

## 2. 当前 50 张稳定基线
- **基线 Commit**：`4489d31 feat: raise native batch limit to 50`
- **Git 状态/工作区状态**：除本设计文档（作为未跟踪/待纳入版本管理文件）外，无产品代码改动，且 Rust processing、batch limit、Tauri 权限均未修改，工作区其他部分完全干净。
- **Feature Flags**：`USE_SIGNAL_GROUPS_FOR_BATTLE` 保持为 `false`。
- **已实现的核心能力**：
  - Native folder picker 权限隔离。
  - Native 预览与 Canvas 串行分析。
  - 物理复制整理 (Physical Org Copy-Only) 的安全防线（防重叠检查、同名规避与全脱敏处理）。
  - 脱敏的 `report.json` 导出与单 PlanId 重复执行防护。
  - Native 模式下 ZIP 导出硬性禁用。
  - Web 与 Demo 回归完整，无任何功能退化。

## 3. 不做事项 (Non-Goals / Constraints)
为了确保扩容过程中的安全与稳定性，以下事项**被严格禁止**：
- **禁止一步到位直接改数**：本轮只做设计，不修改任何产品代码，不直接改 batch limit。后续实施也严禁在无前置安全条件的保障下暴力改数。
- **禁止并发读取**：继续禁止并发读取与并发分析（禁止 `Promise.all` 读取多张原图）。
- **禁止内存占用**：继续禁止 bytes / Blob 进入 React State。
- **禁止全量分析**：继续禁止一次性读取并全量分析整个文件夹的所有文件。
- **禁止 Web Worker**：此阶段及下一阶段继续禁止使用 Web Worker。
- **禁止 SQLite**：此阶段及下一阶段继续禁止使用 SQLite 等数据库依赖。
- **禁止 Native ZIP**：继续完全禁用本地 ZIP 压缩。
- **禁止破坏性整理**：物理整理继续采用 **Copy-Only** 模式，禁止执行任何物理 `move`、`delete` 或 `overwrite` 操作。
- **禁止 Tauri 权限越权**：不新增 Tauri 权限（保持当前的 dialog 等最小限度权限），禁止使用广域文件系统权限 (broad filesystem)、shell 权限或 allow-all。
- **禁止自定义协议越权**：禁止使用或修改资产协议 (assetProtocol / protocol-asset)。
- **禁止联网**：禁止接入任何联网 AI、Supabase、OpenAI API、后端或云端第三方服务（保持纯本地处理）。
- **禁止隐私泄露**：全流程不得显示或输出任何真实的物理路径、文件名、sourceId 或 previewUrl。

## 4. 为什么不能直接实现 100
在 50 张已经稳定的架构下，直接将上限改为 100 会带来以下高危技术风险：
- **垃圾回收滞后与 V8 堆溢出 (OOM)**：
  - 每张图片的 Canvas 像素提取、相似度哈希与质量评分都会产生大量的 ImageData、临时 Canvas 节点和内存碎片。虽然前端已在分析完立即释放，但 JS 虚拟机的垃圾回收 (GC) 是异步非实时的。在 100 次高频串行执行中，GC 响应滞后将导致物理内存暴涨，在第 70-80 张左右极易触发 Tauri 页面 OOM 崩溃。
- **Tauri IPC IO 负载加倍**：
  - 100 次高频调用 `read_native_preview_bytes` 导致渲染进程与主进程频繁交互，增加了数据通道复制和内存调度的消耗。
- **时间成本与无响应体验**：
  - 100 张照片分析耗时通常超过 1.5 分钟。若无明确的进度条百分比展示、跳过数提示以及随时“取消”的机制，用户会处于焦虑和疑惑中，甚至强行关闭程序，破坏数据一致性。

## 5. 100 张前置条件
在推进 100 张扩容前，系统必须具备以下安全支撑：
- **完备的进度管理 UI**：进度条平滑，实时指示 `X / 100`，清晰展示跳过（如损坏、HEIC）和失败的照片数。
- **防卡死的取消机制**：设计可中断的串行机制，随时响应用户“停止分析”的动作。
- **数据保留流转**：用户取消或单张失败后，已分析过的成果不废弃，可直接导入结果页查看。
- **回滚与 Fallback 策略**：保留 50 张的常数配置，若 100 张版导致用户崩溃，可快速一键回滚。
- **完整的测试用例支持**：准备包含大文件（>15MB）、HEIC、格式损坏以及 100+ 张各种混合格式的临时测试资料夹。

## 6. 分阶段路线
- **Phase 1：保持 50 张稳定基线**：当前正式基线，不改动，作为 fallback 兜底。
- **Phase 2：100 张设计准备**：本轮只设计，不修改任何代码，仅形成设计文档。
- **Phase 3：100 张实验实现**：下一阶段实现中，通过实验性的 flag （或只控制常数）在测试分支控制，依然保持串行、不加并发、不加 Worker / SQLite。
- **Phase 4：100 张真实 QA**：利用 30 -> 50 -> 100 分级测试，验证混合格式、跳过、copy-only 输出及 report.json 脱敏，并检测 Web / Demo 回归。

## 7. 取消 / 中断机制设计
为了提供对大批量扫描的良好控制，设计以下取消逻辑：
- **停止按钮**：在分析过程中，分析进度条旁边提供“停止分析”或“取消扫描”按钮。
- **打断逻辑**：在串行循环开始前，增加打断判定：
  - 前端设置一个 `isCancelledRef = useRef(false)`，点击取消时置为 `true`。
  - 串行 `for` 循环中，每次迭代前校验：`if (isCancelledRef.current) break;`。
- **数据保留**：
  - 用户点击“停止分析”后，只停止后续分析队列，**已完成的 M 张照片分析结果予以保留**。
  - 未分析的 100 - M 张图片**自动标记为 `skipped` 并保持默认状态，不得自动标为淘汰候选**。
  - 取消后，允许用户进入 Results 页面查看已完成部分的整理建议。
- **整理与防护**：
  - **取消后不自动启动 Similar Groups 识别。**
  - **取消后不自动启动 A/B PK 流程。**
  - **取消后不自动执行 copy-only physical org 物理复制整理。**
  - 整理结果页中，用户必须手动点击对应的按钮来触发相似组识别、A/B PK，或本地物理整理输出。未分析成功的 skipped 照片在物理整理时将被安全跳过。
- **脱敏提示**：任何因错误或取消产生的提示、通知与日志，必须经过严格脱敏，**绝不显示真实路径、真实文件名、sourceId 或 previewUrl**。

## 8. 内存与性能策略
- **出让主线程时间 (Thread Yielding)**：
  - 为给 V8 垃圾回收和浏览器 UI 绘制留出足够的时间窗，每分析 10 张图，执行一次 `await new Promise(r => setTimeout(r, 100))`。
- **垃圾回收促进**：
  - 在 analyzeImage 的 `finally` 块中立即 revoke 对应 ObjectURL，解除对大图 Blob 的所有引用，加快临时内存的释放。
- **进度平滑控制**：
  - 进度 UI 需明确显示“已分析 X / N”，并提供跳过与失败计数。

## 10. QA 策略
在进行 100 张扩容验证时，必须严格执行以下 QA 步骤与验证机制：

### 10.1 初始状态与命令校验
- **Git 状态检查**：运行 `git status --short` 确认除目标修改外无其他杂乱文件。
- **基线差异校验**：运行 `git diff -- src/lib/config/featureFlags.ts` 确认 feature flags 未被修改。
- **静态代码构建**：运行 `npm.cmd run build` 确保无前端编译、打包错误。
- **代码规范校验**：运行 `npm.cmd run lint` 确保无规范及类型警告。
- **桌面调试环境验证**：运行 `npm.cmd run desktop:dev` 验证 Tauri 及 Cargo 能够成功编译并拉起调试窗口。

### 10.2 分级相册规模测试
- **30 张回归**：选择包含 30 张图片的本地测试文件夹，确认旧规模功能表现完全无误。
- **50 张回归**：选择包含 50 张图片的本地测试文件夹，验证 50 张级别处理的性能。
- **100 张测试**：选择包含 100 张图片的本地测试相册，验证处理流程顺利推进到 100 张上限。
- **超过 100 张测试**：选择包含例如 120 张图片的测试文件夹，确认系统只取前 100 张进行预览及处理，超出部分不进入分析队列，且不全量扫描也不崩溃。

### 10.3 格式与多尺寸边界测试
- **格式类型验证**：准备 `jpg`、`jpeg`、`png`、`webp` 的混合格式文件夹，验证分析完整度。
- **大文件跳过测试**：文件夹中混入大于 15MB 的图片，确认系统能够优雅跳过分析并置为 keep 默认（提供脱敏的已跳过原因），不崩溃。
- **HEIC / HEIF 跳过测试**：混入 `.heic` / `.heif` 图片，确认扩展名安全跳过逻辑生效。
- **照片类型与多尺寸测试**：包含横图、竖图、不同分辨率尺寸、混合比例以及混合清晰度图片的回归测试。

### 10.4 功能模块与隐私性回归
- **Native 核心回归**：验证 Native Results 面板、Native Similar Groups（相似分组）和 Native A/B（PK 流程）正常可用。
- **整理输出验证**：验证本地物理整理 (Native copy-only physical org) 会话生成及冲突规避，检查 `report.json` 导出且确认其完全脱敏（不泄露路径、文件名、用户名等）。
- **Native ZIP 拦截**：确认本地数据源下的 Native ZIP 导出功能仍被逻辑禁用。
- **Web 端功能回归**：
  - 网页端多图上传功能 (Web upload)。
  - 网页端结果页呈现 (Web Results)。
  - 网页端的 ZIP 导出、CSV 及 JSON 清单下载 (Web ZIP / CSV / JSON)。
- **演示包回归**：
  - 演示项目的相似分组 (Demo Similar Groups)。
  - 演示项目的 PK 回合流程 (Demo A/B)。
