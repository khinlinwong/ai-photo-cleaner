# Native 更大批量处理安全设计规划（200 张）：CORE-DESKTOP-NATIVE-BATCH-SCALE-200-DESIGN-1

## 1. 背景
AI Photo Cleaner 桌面端（Native）在之前的演进中已成功将批量处理上限提升至 100 张，并保持了良好的稳定性。然而，为了满足用户对于更大批量（如 200 张）照片一次性整理筛选的实际业务诉求，系统需做出相应的扩容规划。200 张的规模已非单纯修改常量所能承载，其读取量、Canvas 分析频率、内存波峰与用户等待时长翻倍，容易引爆 Chromium 虚拟机的 OOM（堆溢出）以及 IPC 通道拥堵等性能与稳定性风险。

因此，本轮工作**只做设计，不实现代码**，旨在规划出一条确保内存安全、响应平滑、控制自如的 200 张扩容安全方案，以作为后续实施阶段的技术规范。

## 2. 当前 100 张稳定基线
目前，系统已稳定在 100 张的处理规模下：
- **核心限额**：
  - Native 预览扫描限制：`PREVIEW_LIMIT = 100`。
  - Native 核心分析限制：`NATIVE_PROCESSING_MVP_LIMIT = 100`。
- **读取与分析逻辑**：
  - 底层继续采用**单队列串行读取**与**串行分析**模式（queue depth = 1），无任何 Promise.all 并发行为。
  - 单张照片最大字节数限制为 `15MB`，超出此限制将优雅跳过。
  - 遇到 `HEIC / HEIF` 格式时自动安全跳过，以规避 Canvas 解码失败。
- **内存安全**：
  - 原始照片的二进制字节流（bytes）及 Blob 对象在分析完相应帧后立即释放，**绝不进入 React 全局 State**，仅保留脱敏后的质量分析指标 and opaque ID。
- **取消与中断机制**：
  - 在分析队列执行时，可通过“停止分析”按钮进行安全打断。
  - 中断后，**已分析部分的数据（0 到 M 张）予以保留**，允许用户手动跳转查看已分析的建议；未分析的图片状态保持默认的 `keep`，并不自动启动 Similarity Group 或 AB 战役。
- **整理输出机制**：
  - 采用 **Copy-Only 物理整理**，将保留照片拷贝到新生成的 export 会话文件夹下，规避物理 `move`、`delete` 或 `overwrite` 风险。
  - 脱敏的 `report.json` 会随物理拷贝同步输出，且整个处理过程中完全对真实路径、真实文件名进行脱敏显示。
  - 本地模式下的 Native ZIP 导出被逻辑禁用。
- **Web 与 Demo 端**：
  - 纯网页版和演示模式运行正常，未发生任何倒退。

## 3. 不做事项 (Non-Goals / Constraints)
为了绝对保障 200 张扩容的健壮性，以下事项在设计和后续实现中**被严格禁止（Red Lines）**：
- **禁止直接实现 200 张**：本轮只修文档修正，不实现 200 张，不修改产品代码，不修改限制常数。
- **禁止直接实现 300+ / 全量文件夹**：禁止为了盲目扩大数量而直接支持 300+ 张照片或对整个文件夹进行不受限全量分析。
- **禁止并发读取 / 分析**：坚决禁止使用 `Promise.all` 或多通道并发读取/Canvas 绘制，必须保持单队列串行（queue depth = 1）。
- **禁止 bytes / Blob 内存残留**：禁止任何图像二进制数据、ImageData 像素块或 Blob 长期保留在 React 全局 State 中。
- **禁止全量扫描分析**：禁止直接分析整个大文件夹的所有文件。
- **禁止 Web Worker**：此阶段及下一阶段实验中禁止引入 Web Worker 处理。
- **禁止近期引入 SQLite**：实验实现阶段近期禁止引入 SQLite 物理数据库或 IndexedDB。SQLite 仅用于未来架构升级讨论。
- **禁止本地 ZIP 压缩**：继续禁用 Native ZIP 导出。
- **禁止物理删除与移动**：物理整理只允许 Copy-Only，绝对禁止使用任何物理 `move`、`delete` 或 `overwrite` 命令。
- **禁止 Tauri 越权**：禁止申请更广泛的文件系统访问权限（如 shell、allow-all、broad filesystem 等），维持最小 Tauri 权限边界。
- **禁止隐私数据泄漏**：禁止在 UI、console 或日志报告中显示/输出任何真实的文件路径、真实文件名、sourceId 或 previewUrl。
- **禁止上传云端**：必须保持纯本地处理，绝对不得将任何照片或分析元数据上传云端。
- **禁止 Web/Demo 功能退化**：必须确保网页版和 Demo 功能无退化。

## 4. 为什么不能直接实现 200
直接将 `NATIVE_PROCESSING_MVP_LIMIT` 改为 200 会带来以下系统性风险：
1. **Chromium / V8 垃圾回收滞后与堆溢出 (OOM)**：
   - 200 张图片的 Canvas 绘制、哈希特征提取与曝光/清晰度计算会创建大量的 ImageData、临时画布等内存垃圾。虽然我们在代码中分析完会立即释放引用，但 V8 垃圾回收是异步的。当分析频率极高且数量达到 100+ 时，GC 速度往往跟不上内存申请速度，使得物理内存持续攀升，极其容易导致 Tauri 渲染进程崩溃或被系统强制杀死（OOM 堆溢出）。
2. **Tauri IPC 管道过载**：
   - 200 次高频调用 `read_native_preview_bytes` 导致主进程与渲染进程之间的数据拷贝与通信频次翻倍，Tauri IPC 管道和主线程的 IO 负载极易出现间歇性卡顿，影响主界面流畅度。
3. **用户等待焦虑与系统无响应**：
   - 200 张照片分析耗时通常是 100 张的两倍以上（通常在 2-4 分钟左右）。如果没有任何分阶段进度指示或进度卡在某个大文件上，用户会产生“程序已死机”的错觉，强行杀进程会导致临时生成的状态数据丢失。
4. **状态与 UI 渲染负担加倍**：
   - Results 页面的 200 张图片列表渲染、200 张级别的相似重复检测算法（计算时间呈 O(N^2) 级增长）以及 PK (Photo Battle) 的组数大幅增加，将给前端的 React 状态树渲染带来巨大的计算负荷。
5. **真实相机高分照片带来的压力**：
   - 用户导入的大多是相机直出的 10-15MB 单张高分照片，连续解码 200 张会让设备发热、CPU/GPU 占用飙升，使潜在的崩溃风险放大数倍。

## 5. 200 张前置条件
在推进 200 张批量处理前，以下前置机制**必须就绪**：
1. **100 张 Fallback 机制**：必须保留 100 张稳定基线逻辑与对应的常数配置，若 200 张实验版出现异常，可一键切回。
2. **打断与安全停止机制**：原有的“停止分析”打断判定必须严格生效，点击停止时需要有平滑的安全中断，且已完成部分的数据可直接用于查看结果。
3. **更明确的进度统计 UI**：增加更细致的已分析、已跳过（HEIC/超大图）、分析失败的计数指示，给用户完整的信息透明度。
4. **分阶段进度提示 (Phase Hinting)**：
   - 底层在处理队列时，UI 需向用户提示当前处于第几阶段。例如按 50 张一包：
     - "正在分析第 1 阶段 (1–50)"
     - "正在分析第 2 阶段 (51–100)"
     - "正在分析第 3 阶段 (101–150)"
     - "正在分析第 4 阶段 (151–200)"
   - 这不仅给用户明确的心理预期，也为底层架构分段做准备。
5. **取消后的结果保留与控制出让**：
   - 用户取消分析后，不自动进入 Results，不自动识别 Similar Groups / A-B PK / 物理复制，界面保持原位并弹出“已停止，您可以手动查看当前已分析的 X 张结果”提示，将控制权交给用户。
6. **全套 QA 测试相册准备**：必须包含足够数量（至少 220 张）的混合图片（JPG、PNG、WEBP、HEIC、>15MB 超大图、损坏文件），用于做压测和回归。

## 6. 分阶段路线
- **Phase 1：保持 100 张稳定基线**：当前正式版本，不改动，做兜底 fallback。
- **Phase 2：200 张设计准备**：本轮操作，仅产出设计文档修正，不涉及任何代码修改。
- **Phase 3：200 张实验实现**：
  - 后续阶段可选。未来在实施阶段可能引入 `USE_SCALE_200_LIMIT` 实验性开关。
  - **重要限制表述收紧**：是否新增该 feature flag 必须在 200 实现 checkpoint 单独确认，本设计文档不要求立即新增。如果未来新增该 flag，默认必须保持关闭以保留 100 张 fallback，且必须经过完整 QA 回归，不得绕过 QA 直接启用 200 张。
  - 核心逻辑保持**串行、无并发、无 Worker、无 SQLite、无 Native ZIP、无 move / delete / overwrite**。
- **Phase 4：200 张真实 QA**：
  - 依据分级测试大纲进行回归，包含 Web/Demo 回归，以及 30 -> 50 -> 100 -> 200 的规模压测。

## 7. 是否需要分批分析架构
为了在不引入并发、Worker 和 SQLite 的情况下，确保 200 张分析的内存安全，我们针对分批分析架构进行如下设计：
- **继续维持单队列串行（queue depth = 1）**：
  - 讨论：单队列串行是系统内存防线的核心基石。因为并发会导致多个大图数据流在内存中重叠，瞬间将物理内存撑爆。而在 200 张级别下，串行执行完全能满足功能性要求。因此，**Phase 3 实验实现中必须继续采用单队列串行**。
- **引入 UI 与底层的分段 Checkpoint（如每 50 张分段）**：
  - **分批提示**：在界面展示“正在处理第 X 阶段 (A-B 张)”。
  - **强制休眠窗 (Thread Yielding Window)**：串行分析到分段点时（如第 50、100、150 张），底层循环强制挂起 `await sleep(300)` 甚至 `500ms`。这能将 CPU 时间片完全出让给浏览器，使垃圾回收器（V8 GC）能对前一阶段产生的 Canvas 和 Blob 引用进行彻底的垃圾清扫，重置内存占用曲线。
- **内存 Batch Checkpoint（不持久化）**：
  - 在 React Context 内部为已完成的批次（如前 50 张）进行内存 checkpoint 标记。如果在分析后半段（如第 120 张）意外触发异常，前端至少能保留前 50 张的完整有效分析结果，降低用户重复分析的时间代价。
- **取消与暂停决策**：
  - **仅保留“停止分析”机制，不需要引入“暂停 / 继续”按钮**。暂停/继续会极大增加状态机的复杂度（挂起状态 of IO 持有、Tauri 的底层连接状态等），风险太高。因此只需维持“一键停止”并保留已分析结果。
- **远期 SQLite 数据库考量**：
  - 200 张规模依然在 React 前端状态管理限度内，本阶段**严禁引入 SQLite**。
  - 仅作为远期架构升级讨论：若未来需要扩展到 500 到 1000+ 张，全量在前端内存维护状态将不再可行，届时需要考虑在 Tauri 主进程引入轻量 SQLite 或 IndexedDB 进行分批磁盘级持久化，以在页面崩溃时进行断点恢复。

## 8. 取消 / 中断机制要求
- **安全停止点**：用户点击“停止分析”时，前端记录取消信号，并在**当前单张分析结束后安全退出循环**，不能强行杀掉底层正在执行的 IPC 文件读取或 Canvas 渲染，确保文件和内存句柄的安全关闭。
- **数据保留与标记**：已分析的 M 张结果必须写入 React 状态；未处理的 200 - M 张照片保持默认 `keep` 状态，其分类置为“已跳过”，原因标注为“用户中止分析”。**绝不自动将未分析的照片标记为淘汰候选**。
- **控制出让**：中断发生后，进度页面处于“已停止”状态，必须由用户手动点击操作。**绝对禁止自动跳转到结果页，绝不自动开启相似重复计算、AB 对决或物理复制**。

## 9. 内存 / 性能策略
- **出让主线程时间 (Thread Yielding)**：
  - 在串行分析循环中，除了分段 Checkpoint 处的长休眠外，平时每分析 5 张图片，就执行一次短暂休眠 `await sleep(50)`，保障 UI 主线程对用户交互的实时响应。
- **Canvas / Blob 的强力释放**：
  - 在像素读取和 canvas 绘制完成后，立即调用 `URL.revokeObjectURL` 释放对图像 Blob 预览的临时内存占用；并在 `finally` 块中将 ImageData、Canvas 节点等临时引用显式置为 `null`。
- **长列表虚拟滚动 (Virtual Grid)**：
  - 200 张照片在 Results 页面展现时，如果一次性渲染 200 个复杂的 DOM 节点，会导致滚动极其卡顿。Results 页面必须接入 `VirtualPhotoGrid` 虚拟滚动组件，仅渲染视口内的图片节点，将 UI 渲染开销降至最低。

## 10. UI 提示策略
- **本地扫描入口**：
  - 明确文案提示：“桌面端批量分析：当前最多分析前 200 张照片，超过限制的照片将不会被分析”。
- **长耗时提示**：
  - 在进入分析页面时，当分析图片数量超过 100 张，UI 需加挂提示文案：“预计分析需要 2-3 分钟，请保持程序运行，建议您中途不要关闭窗口”。
- **统计透明度**：
  - 进度面板必须实时显示：`已成功分析 X 张`、`跳过 HEIC / 超大图 Y 张`、`失败 Z 张`。
- **手动查看结果**：
  - 分析结束后展示“查看整理结果”按钮，禁止自动页面路由。

## 11. QA 策略
在进入 200 张批量处理实施前，必须规划好全方位的 QA 验证流程：

### 11.1 静态与命令校验
- `git status --short`：确认工作区干净，仅有修改的设计文档。
- `git diff -- src/lib/config/featureFlags.ts`：确认 `USE_SIGNAL_GROUPS_FOR_BATTLE` 保持为 `false`。
- `npm.cmd run build`：确保编译无报错。
- `npm.cmd run lint`：确保 lint 检测通过，无警告。
- `npm.cmd run desktop:dev`：确认 Tauri 调试环境成功启动。

### 11.2 分级相册规模压测
- **30 张回归**：验证小批量常规整理。
- **50 张回归**：验证中批量常规整理。
- **100 张回归**：验证当前稳定上限下的性能表现。
- **200 张测试**：选择 200 张图片的真实相册，执行完整的读取和分析。
- **超过 200 张测试（如 220 / 250 张）**：选择超过 200 张的文件夹，验证系统仅截取前 200 张，多余的照片不会被读入或分析，且界面正常不崩溃。

### 11.3 格式与尺寸边界测试
- **混合格式测试**：包含 `jpg`, `jpeg`, `png`, `webp` 等多种常见格式。
- **HEIC / HEIF 跳过验证**：确认 HEIC 图片在分析队列中被安全跳过（标记为已跳过且格式不支持，默认 status 为 keep，不崩溃）。
- **超大图跳过验证**：混入大于 15MB 的图片，确认系统在 `read_native_preview_bytes` 阶段安全拦截并跳过。
- **尺寸与排版**：包含横图、竖图、混合长宽比的回归。

### 11.4 取消与异常流测试
- **中途停止测试**：在分析到第 30 张、80 张、150 张时分别点击“停止分析”，确认已分析的图片被保留并可手动查看，未分析的图片标记为已跳过且不自动路由，不自动识别相似重复。
- **全量分析不取消测试**：完整跑完 200 张分析，监测系统的物理内存波峰，确认无堆溢出（OOM）崩溃。

### 11.5 功能回归校验
功能回归必须包含且单独列出：
- **Native Results**：确认 200 张或部分取消后的结果页可正常显示、滚动、统计。
- **Native Similar Groups**：确认用户手动触发后正常识别，不自动启动。
- **Native A/B**：确认用户手动触发后 Photo Battle 正常，不自动启动。

### 11.6 安全性与 Web/Demo 回归
- **物理复制验证**：物理整理拷贝仅在生成的 session 下创建 Keep 和 Cull-Candidates 文件夹并拷贝文件，绝对没有 `move`、`delete` 或 `overwrite` 逻辑，确认同名规避生效。
- **脱敏 report.json**：验证物理整理输出的 `report.json` 中不出现任何真实的物理路径和真实文件名。
- **ZIP 导出限制**：确认 Native 状态下，ZIP 导出按钮确实被安全逻辑禁用。
- **Web / Demo 兼容性**：验证网页端的上传、结果展示、ZIP/CSV/JSON 导出回归；验证 Demo 的相似组 PK 流程未发生倒退。

## 12. 回滚 / fallback 策略
- **回滚常数保留与 Feature Flag 规约**：
  - 在 `PhotoWorkspaceContext.tsx` 中必须保留 `NATIVE_PROCESSING_MVP_LIMIT = 100`。
  - **重要限制表述收紧**：是否新增 feature flag 或实验开关必须在 200 实现 checkpoint 单独确认，且本设计文档不要求立即新增。如果未来新增，该开关默认必须保持关闭，以保留 100 张 fallback，且不得绕过 QA 直接启用 200 张。
  - 若 200 张实验版在大规模测试中出现 OOM 或性能崩溃，可通过关闭 feature flag 或恢复常数一键切回 100 张或 50 张的稳定基线，提供无缝的回滚通道。
