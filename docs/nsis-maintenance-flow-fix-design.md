# NSIS 安装包维护与重装流程修复设计：CORE-DESKTOP-NATIVE-NSIS-MAINTENANCE-FLOW-FIX-DESIGN-1

## 1. 当前复现问题摘要

在前序的手动复现矩阵测试中，已完全锁定并重现了以下现象与结论：
1. **产品与卸载单体逻辑正常**：App 业务代码不干涉卸载；单独通过 Windows Apps 或控制面板运行 `uninstall.exe` 能干净彻底地删除所有文件和注册表项，随后正常退出关闭。
2. **重装/覆盖安装循环问题**：在系统已安装软件的状态下，再次运行 NSIS `setup.exe`。如果用户在安装器的重装提示（`PageReinstall`）中选择了“卸载现有版本”：
   - 安装器会调用旧版本的 `uninstall.exe` 执行静默清理。
   - 卸载执行完毕后，主安装器进程 `setup.exe` 没有退出，而是自动流转到下一个安装页面（`PageInstFiles`），重新把文件写入，导致用户以为软件被装回。
3. **MSI 方案存在权限壁垒**：MSI 包有效且卸载无循环，但因其写 HKLM 注册表需要强制管理员权限，在普通非管理员权限下运行 silent 安装会直接抛出 `1603` 权限不足错误。普通用户的 Alpha 分发包仍需以 NSIS `currentUser` 为主要方向。
4. **测试警告**：目前测试人员和内部用户严禁混用 NSIS 和 MSI 安装包，防止因物理目录重叠与注册表混杂导致系统卸载入口损坏。

本方案旨在针对“已安装状态下运行 `setup.exe` 选择卸载后会自动重装”这一核心痛点进行精细化的安装器行为和流程设计。

## 2. 当前 NSIS 默认流程诊断

### 2.1 默认 NSIS 重装逻辑推测
Tauri 默认的 NSIS 模板（基于 Handlebars 注入生成）中，当 `setup.exe` 运行时，它会执行以下维护/重装检测流程：
1. **检测现有安装**：在 `.onInit` 或 `PageReinstall` 的首要阶段，通过 `ReadRegStr` 读取当前用户的 `Uninstall` 注册表键值，确认是否已安装。
2. **加载 Reinstall 页面**：如果已存在，弹出带有单选按钮或提示的选择界面，选项一般包括：
   - 卸载现有版本并重装（Uninstall first）。
   - 覆盖安装/修复（Add/Reinstall）。
3. **执行卸载**：当用户选择卸载旧版本时，主安装程序 `setup.exe` 通过 `ExecWait` 调用旧版卸载器：
   `ExecWait '"$INSTDIR\uninstall.exe" /S _?=$INSTDIR'`
4. **控制流缺陷**：`ExecWait` 执行完毕后，`uninstall.exe` 进程退出。主安装器进程 `setup.exe` 重新获取控制权，但由于没有检测“用户此时意图是否仅仅是卸载”，或者在卸载分支结束后**缺失了主进程终止指令（如 `Quit` 或 `Abort`）**，主安装器继续走后续的安装和文件写入页面，形成了非预期的重装。

## 3. 可选修复方案比较

针对上述控制流缺陷，评估以下四种方案的优劣与可行性：

### 方案 A：只在配置中固化 `installMode: "currentUser"`
- **描述**：仅在 `tauri.conf.json` 中配置显式的 `currentUser` 安装模式，规范化安装路径。
- **优点**：零开发成本，隔离了多用户及 HKLM 管理员权限冲突。
- **缺点**：**完全无法解决**“卸载后继续安装”的维护流程问题。该配置仅用于固化用户态路径，无法改变 NSIS 脚本生成的控制流逻辑。
- **结论**：作为基础配置固化项，不作为流程解决方案。

### 方案 B：使用 `installerHooks` 挂载外部脚本阻断
- **描述**：编写 `nsis-hooks.nsh` 文件，配置到 `tauri.conf.json` 的 `installerHooks` 中。
- **生命周期分析**：
  - Tauri 的默认 NSIS 支持 `NSIS_HOOK_PREUNINSTALL` 和 `NSIS_HOOK_POSTUNINSTALL`。
  - **严重风险**：这些卸载钩子（Pre/Post Uninstall）是在**卸载器进程 (`uninstall.exe`) 的上下文中执行的**。如果我们在 `NSIS_HOOK_PREUNINSTALL` 中直接注入 `Quit` 脚本，会使卸载器本身提前中止，导致文件和注册表完全没有删除；如果写在 `NSIS_HOOK_POSTUNINSTALL`，它也只会在卸载器子进程结束前执行。
  - **技术壁垒**：子进程 `uninstall.exe` 无法将 `Quit` 状态冒泡回正在用 `ExecWait` 等待的父进程 `setup.exe`。因此，`installerHooks` 无法以安全的方式阻断父安装器的后续安装。
- **结论**：**不建议采用。此方案技术上不可行且极易引发卸载不完整或残留**。

### 方案 C：使用 `custom NSIS template` 进行流程级定制 (推荐)
- **描述**：基于 Tauri 官方默认的 `installer.nsi` 模板创建自定义模板，仅在关键的 `PageLeaveReinstall` 逻辑中加入退出判断。
- **设计细节**：
  1. 识别检测已安装版本的选择页面（如 `PageReinstall`）。
  2. 当用户选择“仅卸载”或“卸载旧版本”并完成 `ExecWait` 调用后，在 `PageLeaveReinstall` 中判定卸载已结束。
  3. 执行完卸载后，**立即在 `setup.exe` 主进程中调用 `Quit`**。
  4. 只有当用户明确选择“覆盖安装/修复”时，才绕过卸载或在卸载后继续向下流转。
- **优点**：对父安装器控制流最可控，能彻底解决“卸载后重装”的循环。
- **缺点**：维护成本较高。需要手动拷贝并维护一个 `.nsi` 模板，后续需要跟随 Tauri CLI 版本更新同步维护，防止与新版 Tauri 属性冲突。
- **结论**：**这是技术上唯一能够彻底闭环解决此问题的方案**。后续实现应优先基于此方案开展最小化改动设计。

### 方案 D：短期 Alpha 规避策略 (零开发风险)
- **描述**：不改动任何安装包打包配置或模板逻辑。仅通过 Release Notes、测试文档和操作提示明确警告测试人员与早期内测用户：
  - **不要运行 setup.exe 试图进行卸载**。
  - **卸载必须且只能通过“控制面板”或“Windows 设置 -> 应用 -> 安装的应用”触发**。
  - `setup.exe` 安装包仅用于首次全新安装或覆盖安装。
- **优点**：零开发成本，无任何安装包崩溃或注册表残留风险。
- **缺点**：无法从根源上阻止普通用户的误操作，用户体验不完美。
- **结论**：在正式发布前，此方案可作为内测和开发阶段的规避底线。

## 4. 推荐方案倾向

我们应采取**稳健、克制、循序渐进**的策略，避免盲目重写模板导致打包失效：

1. **短期 Alpha 规避（当前立即执行）**：
   - 内部测试期间，所有测试人员一律仅通过 Windows Apps / Settings 卸载应用。
   - 并在发布说明中明确注明该操作约束。
2. **第一推荐（中长期核心修复方案）**：
   - 在后续的实现 Checkpoint 中，设计并引入 **最小改动的 Custom NSIS template**。
   - 提取当前 Tauri v2 版本所对应的默认 `installer.nsi`，仅在 `PageLeaveReinstall`（或处理卸载选择的 Leave 函数）中，在卸载逻辑执行完毕后追加 `Quit`，使安装包在完成卸载职责后立即安全关闭。
   - **本轮只做设计，严禁直接实现**。
3. **第二推荐（辅助配置固化）**：
   - 将 `installMode: "currentUser"` 写进 `tauri.conf.json` 作为常规用户态打包规范。
4. **禁用项**：
   - **严禁直接在 hooks 中写入 `Quit` 实现阻断**（因为会导致卸载器未删文件先中断）。
   - **严禁直接在业务层代码（TS/Rust）中执行文件系统级的卸载动作**。

## 5. 安全边界与红线

所有修复方案均不得触碰或越过以下安全红线：
- **隔离业务与安装器**：App 运行时代码（业务逻辑）绝不处理任何 uninstall.exe 的释放、路径擦除或文件自删。
- **最小权限约束**：不开启 Tauri 的广域 shell 或 fs-allow-all。不因安装器问题申请更高级别的系统特权。
- **原件只读**：安装与卸载逻辑与用户的任何照片物理目录隔离，无论安装器如何动作，绝不改动、删除或触碰用户导入的照片数据。
- **禁止混用**：开发与测试管道不得混合分发 MSI 和 NSIS，防止在 `%LOCALAPPDATA%` 重合路径下造成双重卸载项残留。
- **Native 安全特性保留**：本地模式下的 report.json 脱敏、copy-only 纯复制以及 ZIP 按钮安全置灰 guard 机制必须原样固化。

## 6. 验收测试清单

后续完成实现后，必须在干净的虚拟机/Sandbox 中完成以下 11 项完整的验收用例：

1. **TC-01 NSIS 全新安装**：在无软件残留机器上安装，正常完成并打开 App。
2. **TC-02 已安装下再次运行 setup.exe**：确认是否正确弹出维护/重装选项，不能直接静默执行。
3. **TC-03 setup.exe 维护页选择卸载**：**重点验证**，在点击卸载并清理完文件后，安装包窗口应立即退出关闭，绝不进入重装流程。
4. **TC-04 setup.exe 维护页选择覆盖安装/修复**：确认能够正常覆盖更新，版本及文件更新正确。
5. **TC-05 Windows Apps / Settings 卸载**：确认通过系统入口卸载依然完全正常，且卸载后不会重新安装。
6. **TC-06 卸载后重装**：先卸载再全新运行安装，无残留配置引起的异常。
7. **TC-07 卸载入口唯一性**：安装后，Windows 卸载列表中只出现唯一的“AI Photo Cleaner”条目，无重复入口。
8. **TC-08 App 启动验证**：安装及覆盖安装后，App 的桌面快捷方式和主进程均能正常拉起并离线工作。
9. **TC-09 照片目录完整性**：安装、覆盖和卸载过程中，测试照片源目录内的图片完好无损。
10. **TC-10 功能组件闭环**：验证 Results 页、Similar Groups 模块、A/B 模块及 Copy-only 整理功能不受打包方式影响。
11. **TC-11 警告行为记录**：在 Sandbox 下截图并记录未签名状态下的 SmartScreen 拦截阻断提示，作为内测文档说明素材。
