# AI Photo Cleaner 核心稳定化下一阶段规划 - CORE-STABILIZE-NEXT-STEPS-PLANNING

## 一、 当前已完成成果

在核心稳定化的前几个阶段中，系统已成功实现了以下关键成果：
1. **VirtualPhotoGrid 挂载成功**：Results 页面已全面接入自研、自适应的无状态虚拟滚动网格，解决了大列表照片卡片在 DOM 树中节点过密、内存积压的卡顿痛点。
2. **算法 Parity 多轮验证通过**：相似检测 `duplicate` 模块中新客观 `signal groups` 转换机制在 `USE_SIGNAL_GROUPS_FOR_BATTLE = true` 临时分支下通过了 100/200/300 张多格式混合文件的 Parity 一致性比对，Leader 错配数为 0。
3. **稳定 QA 摘要输出**：实现了 development-only 双条件卫哨拦截的全局 `window.__AI_PHOTO_CLEANER_QA__` 属性注入，支持自动化脚本比对，不影响主流程和打包性能。
4. **生产环境绝对隔离**：经生产构建验证，production 状态下 `window.__AI_PHOTO_CLEANER_QA__` 属性处于未定义（`undefined`）状态，实现了无污染阻断。
5. **100 / 300 张相册感样本 parity 对齐**：模拟日常手机真实连拍、模糊、纵横比各异的测试样本均在当前验证中顺利跑通。
6. **大图全流程算法对齐稳定**：100 张和 200 张 3MB-10MB 大尺寸 JPG 客观分析、双路 Parity 校验及 Photo Battle 状态机在压力测试下表现稳定。
7. **分批 ZIP 导出机制确立**：放弃前端单 Blob 一次性压制大包的激进方案，完成了串行打包、下载延迟间隔触发的多 part 导出，且对中等压力（如 100 张大图）表现稳定。
8. **分批参数调优收敛**：结果页面常量已收缩为更温和保守的 `300MB` / `30 张` / `3000ms`。
9. **Results 页面提示与容灾引导完成**：在 results 页面成功集成了常驻轻提示、isZipping 动态等待提示、基于 photos.length 动态计算的二级照片数量警告，以及 `catch` 异常路径下的内联黄色警告框。文案中统一使用“淘汰候选”，废弃“删除”。
10. **特性开关锁定**：`USE_SIGNAL_GROUPS_FOR_BATTLE` 默认值在物理上强力保持为 `false`，生产环境 100% 锁定于 legacy 稳定底座。

## 二、 当前明确边界

为降低系统性风险，现阶段我们划定了清晰的系统与原型边界：
1. **200 张大尺寸 JPG 导出中断（物理边界）**：200 张大尺寸 JPG 淘汰区 + 保留区多 part ZIP 连续写入依然可能触发 `DownloadInterrupted` 错误。这已被确认为网页端 Blob 导出及磁盘 I/O 并发的浏览器底层黑盒边界，不应在当前 Next.js 浏览器原型下继续盲调 ZIP 参数。
2. **200 张大图导出不作为原型硬性指标**：承认浏览器原型的导出能力天花板，在考虑公开 Beta 阶段前，必须先在 results 页面上完成清晰的容量限制提示、失败引导与分批导出 UX 改造（此项已在 `CORE-ZIP-EXPORT-UX-LIMIT` 阶段作为 beta 前置条件完成）。
3. **多媒体格式覆盖限制**：HEIC 格式及相机 RAW 格式在当前扫描及分析层依然被主动物理隔离，不予以处理。
4. **真实客户相册盲区**：目前所有测试均在本地预设测试集下完成，没有在真实用户的复杂环境、巨型相册数据上进行过长期持久比对。
5. **多端及弱环境覆盖缺失**：低配置设备、非主流浏览器的长期运行稳定性仍缺乏检验。
6. **Production 零污染防线**：`USE_SIGNAL_GROUPS_FOR_BATTLE` 严禁默认开启为 `true`，且不移除 legacy detectDuplicates 稳定路径。

## 三、 下一阶段候选路线评估

根据当前成果与剩余边界，我们梳理出以下五条可选的后续技术路线：

### 路线 A：Core UI / Product Polish (结果页面细节润色)
- **目标**：优化 results 页面的层次可读性，进一步检查文案、小屏自适应、各按钮禁用状态、无照片时的空状态（Empty States），彻底改善用户的使用和理解成本。
- **优点**：风险极低，不改动任何核心算法和 ZIP 导出底座，可显著改善用户进入 beta 前的第一感官体验。
- **优先级**：**高**。

### 路线 B：Photo Battle 体验稳定化 (擂台对决体验)
- **目标**：深度检查 Photo Battle 流程中的“保留左图”、“保留右图”、“两张都保留”、“两张都淘汰候选”、“跳过/Skip”以及“安全关闭（Esc）”等各条物理链路；在大组连拍照片多对局时，防止用户出现交互迷失；优化对局内文案。
- **优点**：直接关系到系统整理照片的最核心体验，能有效降低误操作率。
- **优先级**：**高**。

### 路线 C：Repeatability / Long-run Stability (重复运行稳定性持续验证)
- **目标**：不继续扩大单次物理压测（不强求 200 张大包一次性全部通过），改为对 100 张或中等数据集进行多轮重复的完整生命周期测试，重点监控内存平稳度、句柄销毁与 reset 后的状态积压。
- **优点**：能进一步打磨系统在长时间重复运行下的高吞吐能力。
- **优先级**：**中**。

### 路线 D：HEIC / RAW 格式支持可行性调研
- **目标**：不做直接代码实现，仅对苹果 HEIC/RAW 格式在浏览器端的高效解码、格式转换（如 Canvas 像素解析）及 Tauri 桌面端的原生适配路线进行可行性调研。
- **优点**：这是产品走向真实相册所必不可少的一步。
- **优先级**：**中低**。

### 路线 E：ZIP Worker / Stream / Tauri Export (大导出根治方案)
- **目标**：规划 Web Worker 异步打包、File System Access API 流式压缩写入、以及 Tauri 后端原生 Rust 文件复制的架构方案。不在 Next.js 前端原型中直接编码开发。
- **优点**：是解决 200+ 张大图 ZIP 导出 DownloadInterrupted 的最终根治方案。
- **优先级**：**暂缓**（未来重构阶段再行启动）。

## 四、 推荐下一步方向：CORE-RESULTS-UX-POLISH-PLANNING

基于目前的稳定化进程，**我们强烈推荐下一阶段推进：CORE-RESULTS-UX-POLISH-PLANNING**。

### 推荐理由：
1. **算法与 I/O 边界已收敛**：相似组双路 Parity 一致性在多格式下已经稳固建立，200 张 ZIP 限制已通过 UX 限制提示成功规避，继续深挖底层开发或盲调参数已无边际收益。
2. **急需提升用户理解度**：AI Photo Cleaner 在进行 Beta 测试前，最紧迫的需求是确保用户对结果页面中的“保留区”与“淘汰候选区”口径有绝对清晰的感知，同时防止因为布局挤占或提示过密导致操作困扰。
3. **成本低且见效快**：Results 页面的 UI 优化属于纯表现层适配，风险极低，最适合在当前核心逻辑稳定后作为第一优先级铺展，确保 MVP 演示体验足够 premium。

---

## 五、 CORE-RESULTS-UX-POLISH-PLANNING 具体规划方向

在接下来的 UI polish 阶段，应优先检查并落实以下八个维度：
1. **UI 元素层级梳理**：检查 Results 页面的标题、顶部统计栏（保留区/淘汰候选区的张数与文件估算体积大小）、安全策略卡片以及 ZIP 导出按钮区是否排布清晰、易于识别。
2. **文案口径统一**：确保页面各处以及 dialog 中关于“保留”、“淘汰候选”的表述 100% 保持一致，坚决杜绝“删除”、“移出”等产生 SaaS 理解歧义的脏文案混用。
3. **ZIP 提示间距优化**：评估本次实现的常驻轻提示、照片数量警告、isZipping 提示以及 catch 警告在垂直排列时是否会显得过密，优化间距和边距。
4. **自适应布局微调**：检查窄屏和模拟移动端设备上的自适应表现，微调卡片和网格的 `padding`。
5. **照片列表空状态 (Empty States) 适配**：设计当保留区或淘汰候选区照片数为 0 时的页面显示（例如显示柔和的 icon 和“暂无保留/淘汰照片”字样），防止页面出现空秃。
6. **Photo Battle 入口清晰化**：如果仍有未处理的相似照片，检查 results 卡片中的“继续 PK 对比”按钮是否足够显眼，以强力引导用户完成核心整理。
7. **按钮状态自洽性**：检查在 isZipping、导出区域为空以及 pendingGroups 存在时，各个按钮的 hover 效果、激活态及 `disabled` 样式是否符合常理。
8. **开发红线**：**不修改聚类与客观分析算法，不修改可见性分类划归，不修改 ZIP 分批打包核心逻辑。**

---

## 六、 明确暂缓事项

我们将在未来较长一段周期内，严守以下红线，明确**不进行**以下开发事项：
1. **不进入 beta 阶段**：系统在 results UI polish、Photo Battle 细致体验打磨及多轮重复性深度回归前，坚决不启动 beta。
2. **USE_SIGNAL_GROUPS_FOR_BATTLE 不设为 true**：生产配置默认继续锁死为 `false`，生产主流程绝不启用该灰度分支。
3. **不移除 legacy 稳定底座**：继续全量保留 detectDuplicates 稳定链路，防范非预期 regression。
4. **不继续盲调 ZIP 参数**：开发环境的 parameters 常量不再改动。
5. **不继续强跑 200 张大尺寸 JPG 的 ZIP 导出测试**：承认浏览器 Blob 导出能力天花板，防止 OOM 与管道堵塞。
6. **不实现 Worker、流式 ZIP 和 Tauri 桌面物理操作**：相关大重构全部推迟到后续桌面客户端迁移阶段。
7. **不实现 HEIC 和 RAW 格式转换**。
8. **不引入新依赖**。
9. **不改写用户决定的二值分类逻辑**。
