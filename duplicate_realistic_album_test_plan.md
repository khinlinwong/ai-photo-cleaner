# AI Photo Cleaner 真实相册感样本灰度测试规划 - CORE-DUPLICATE-REALISTIC-ALBUM-PLANNING

## 一、测试目标
本轮测试的核心目标不是单纯扩大样本集规模（例如达到 500+ 张图片），而是着重提高测试样本的真实性，使其更贴近用户日常手机相册的使用场景。

### 具体目标：
1. 构建 100-300 张更接近真实手机相册结构、特征的非隐私测试集。
2. 验证 signal groups 分支在更真实相册结构下的 parity（新旧算法分组期望一致）。
3. 验证 Photo Battle 功能在更复杂的相似组下依然能正常交互与流转。
4. 验证 ZIP 二值导出功能在包含各类真实照片时依然正常运行。
5. 验证 `/results` 页面在使用虚拟滚动网格（VirtualPhotoGrid）时的平滑度与稳定性，无白屏或浏览器卡死。
6. 验证 `window.__AI_PHOTO_CLEANER_QA__` 全局变量在各种测试状态下均能稳定读取并输出期望的 parity 指标。
7. 确保测试完毕后，系统配置恢复为 `USE_SIGNAL_GROUPS_FOR_BATTLE = false`，不影响生产环境。

---

## 二、测试图片来源与边界
为了保障数据安全与隐私，测试过程中使用的图片必须遵循严格的合规性要求。

### 允许使用的来源：
1. **个人拍摄的非隐私、非敏感照片**：例如自己在户外拍摄的普通风景、绿植、地标建筑等。
2. **无隐私信息的客观实体照片**：无人物面部、无车牌号、无任何可识别的个人或家庭住址信息的普通风景/物品/建筑物照片。
3. **可公开使用的无版权或测试用途图片**：从 Pixabay、Unsplash 等合规渠道下载的无人物敏感信息的公版图片。
4. **人工生成或修改的、符合真实相册特征的图片**：
   - 针对同一张图进行不同的曝光度/对比度调整。
   - 针对同一张图进行轻微的高斯模糊或抖动处理。
   - 拍摄同一物品在连续微调角度下的多张照片（模拟相似构图/连拍）。
   - 截屏、UI 界面图（无敏感聊天记录、无账户/密码信息等）。
   - 经过压缩成不同质量的 JPG 图片。

### 严禁使用的来源：
1. **任何客户照片或真实用户照片**。
2. **包含家庭成员、个人隐私或人脸清晰的照片**。
3. **包含车牌、证件、地址、发票账单、收据的照片**。
4. **包含聊天记录、银行账户、护照、签证等极为敏感的系统或屏幕截图**。
5. **任何其他无法公开、可能侵犯隐私或带有机密性质的内容**。
6. **严禁将任何测试图片放入项目目录内部**。

## 三、测试集组成规划
针对 100 张和 300 张两个测试档位，规划不同的相册结构组成，以最大限度还原真实相册的分布特征。

### 1. 建议 100 张档位组成：
- **连拍 / burst 风格 (20张)**：针对同一场景进行轻微位移、角度变化的连续照片，形成 4-5 个相似组。
- **截图 / UI 图风格 (15张)**：系统界面、应用界面的截图，用于测试低熵/高边缘特征下的匹配。
- **模糊或轻微抖动图 (15张)**：焦距未对准、手抖导致轻微模糊的图片，与清晰图混杂。
- **曝光异常图 (15张)**：过亮（曝光过度）、过暗（曝光不足）、低对比度等照片。
- **普通独立照片 (15张)**：孤立的风景、日常物品，预计不会被判定为任何相似组。
- **不同尺寸 / 横竖比例照片 (10张)**：包含裁剪过的、长图、宽图，测试尺寸差异下的算法鲁棒性。
- **WebP / PNG 混合图 (10张)**：除 JPG 外的非主流相册格式图片。

### 2. 建议 300 张档位组成：
- **连拍 / burst 风格 (60张)**：多组不同场景下的连续拍摄，每组 5-10 张。
- **截图 / UI 图风格 (45张)**：各类无敏感信息的网页截图、应用界面截图。
- **模糊或轻微抖动图 (45张)**：不同程度模糊、失焦、抖动的混合照片。
- **曝光异常图 (45张)**：各种复杂光线环境、夜景、逆光照片。
- **普通独立照片 (45张)**：各自独立的普通风景、静物照。
- **不同尺寸 / 横竖比例照片 (30张)**：各类不同分辨率及非标准比例的照片。
- **WebP / PNG 混合图 (30张)**：更大量的 PNG 和 WebP 图片。

### 格式比例分配建议：
- **JPG**：约 70%
- **PNG**：约 20%
- **WebP**：约 10%

> [!WARNING]
> **暂不纳入本轮测试的格式**：
> - HEIC（由于浏览器原生支持及转换逻辑差异，暂不处理）
> - RAW / DNG（相机原始数据，体积过大且暂不支持）
> - 视频文件
> - GIF 动图

## 五、测试步骤
在后续进入执行阶段时，必须严格按照以下步骤开展测试：

1. **准备阶段**：
   - 确认当前工作区 `git status` 是干净的，没有未提交的代码或图片。
   - 确认 `src/lib/config/featureFlags.ts` 中 `USE_SIGNAL_GROUPS_FOR_BATTLE` 默认值为 `false`。
2. **开启开关**：
   - 临时修改 `src/lib/config/featureFlags.ts`，将 `USE_SIGNAL_GROUPS_FOR_BATTLE` 修改为 `true`。
3. **运行服务**：
   - 启动本地开发服务器：`npm run dev`。
4. **导入测试**：
   - 打开浏览器，进入 `/desktop`。
   - 导入已规划好的 100 张外部测试照片。
5. **处理与加载**：
   - 观察 `/processing` 页面的处理进度，确保所有照片成功分析。
6. **指标提取**：
   - 进入 `/results` 页面。
   - 打开浏览器控制台，读取并复制出 `window.__AI_PHOTO_CLEANER_QA__` 的完整 JSON 输出。
   - 记录新旧算法的分组匹配度。
7. **功能验证**：
   - 测试 **Photo Battle** 功能，在相似组较多、较复杂的情况下，验证其卡片切换、操作（保留左/保留右/两张都保留/都标记为淘汰候选/跳过/重置）是否期望一致。
   - 测试 **ZIP 二值化导出**，验证最终下载的压缩包里分类是否正确，且与界面展示分区期望一致。
   - 在 `/results` 页面反复上下滚动，观察 VirtualPhotoGrid 的滚动平滑度及内存占用情况。
8. **升级测试**：
   - 如果 100 张样本测试完全通过，再将外部测试集扩展到 300 张，重复上述第 4 至 7 步。
9. **恢复环境**：
   - 测试结束后，将 `src/lib/config/featureFlags.ts` 中的 `USE_SIGNAL_GROUPS_FOR_BATTLE` 恢复为 `false`。
   - 运行命令 `git diff -- src/lib/config/featureFlags.ts`，确认没有任何修改遗留。
   - 运行项目构建和代码检查：`npm run build` 和 `npm run lint`，确保项目依然健康。
   - 再次检查 `git status`，确保没有多余的测试图片、下载的 ZIP 或临时日志留在项目目录中。

## 六、记录指标与数据结构
在测试过程中，必须对每一档测试记录以下三类指标：

### 1. 基础指标
- **Total Images**：测试导入的图片总数。
- **Format Breakdown**：JPG / PNG / WebP 的具体数量。
- **Total File Size**：测试集总文件大小（MB）。
- **Processing Time**：从导入开始到 `/results` 渲染完毕的耗时。
- **Results Render Status**：`/results` 页面是否能正常渲染，是否有异常白屏或卡死。
- **Grid Scroll Performance**：虚拟网格滚动是否顺畅，是否有明显的渲染抖动。
- **Browser Responsiveness**：测试过程中浏览器是否出现“无响应”或崩溃提示。

### 2. QA Parity 指标
通过 `window.__AI_PHOTO_CLEANER_QA__` 获取的 JSON 结构，记录以下数据：
- `oldSimilarGroupCount`: Legacy 分组模式下的相似组总数。
- `newSimilarGroupCount`: Signal 分组模式下的相似组总数。
- `similarGroupCountMismatch`: 是否存在相似组数量不一致（期望为 `false`）。
- `oldSimilarGroupedPhotoCount`: Legacy 分组模式下的相似照片总数。
- `newSimilarGroupedPhotoCount`: Signal 分组模式下的相似照片总数。
- `similarGroupedPhotoCountMismatch`: 是否存在相似照片总数不一致（期望为 `false`）。
- `leaderMismatchCount`: 代表图（Leader）不一致的组数量（期望为 `0`）。
- `generatedAt`: 数据生成的时间戳是否存在。
- `source`: 确认数据来源为 `'duplicateGroupQA'`。
- **安全检查**：检查该 JSON 是否仅包含比对指标，不包含任何图片 Base64 或敏感的本地路径。

### 3. 业务功能指标
- **Photo Battle 触发**：进入相似组时，Photo Battle 交互界面是否能够自如展现与隐藏。
- **单边保留交互**：点击“保留左边”或“保留右边”后，是否正确将对侧标记为淘汰候选，并能够继续下一张。
- **全部保留/全部淘汰**：点击“两张都保留”或“都标记为淘汰候选”后，页面状态是否与预期一致。
- **跳过与重置**：点击“跳过”是否进入下一对，点击“重置”是否能清空当前相似组的决策。
- **ZIP 导出一致性**：导出的 ZIP 压缩包内“保留”和“淘汰候选”分区是否和页面上的最终勾选状态期望一致。
- **最终分类限制**：是否保持“保留”和“淘汰候选”的二值分类，不出现第三种最终分类。

## 七、测试通过标准
只有当所有测试指标完全满足以下条件时，方可判定该测试档位“通过”：

1. `oldSimilarGroupCount === newSimilarGroupCount` （分组数量在当前样本下对齐）。
2. `oldSimilarGroupedPhotoCount === newSimilarGroupedPhotoCount` （被分组的照片总数在当前样本下对齐）。
3. `similarGroupCountMismatch === false` （无分组数量差异，期望一致）。
4. `similarGroupedPhotoCountMismatch === false` （无照片数量差异，期望一致）。
5. `leaderMismatchCount === 0` （每组的代表图在当前样本下对齐）。
6. Photo Battle 业务流程完整无阻塞，状态流转在本轮测试中通过。
7. ZIP 导出压缩包结构与文件划分符合二值分类，无非预期遗漏。
8. `/results` 页面在虚拟滚动下性能良好，本轮回归中未发现明显掉帧，无白屏或无响应。
9. 测试完成后，代码完全恢复，`git status` 干净，无任何测试残留。
10. 不影响 production 生产环境（其依然强制运行 legacy 分组）。

## 八、失败处理与容灾预案
如果测试未通过，需采取以下防范和记录措施：

### 触发失败的场景：
1. 分组数量或相似照片总数不一致（Mismatch 为 `true`）。
2. 代表图发生偏移（`leaderMismatchCount > 0`）。
3. Photo Battle 状态异常死锁、按钮无响应或计算出错。
4. 导出的 ZIP 损坏、文件分类遗漏，或分类偏离了二值设计。
5. 页面产生第三种无法归类的分类状态。
6. 浏览器卡顿时间超过 3 秒，或弹出“页面无响应”警告。
7. 测试图片被误复制进项目目录并被 Git 追踪。
8. 无法将 `USE_SIGNAL_GROUPS_FOR_BATTLE` 恢复为 `false`。

### 失败后的处理步骤：
1. **立即停止测试**，不要尝试在测试中途修复或修改 `src` 代码。
2. **恢复 feature flag 开关**：先将 `USE_SIGNAL_GROUPS_FOR_BATTLE` 设回 `false`，确保本地服务退回安全状态。
3. **保存现场数据**：
   - 截图保存出问题的 UI 状态或控制台错误日志。
   - 将 `window.__AI_PHOTO_CLEANER_QA__` 中不一致的具体 JSON 数据保存至项目外的临时文本中。
4. **代码及仓库审查**：
   - 运行 `git diff` 确认只有 feature flag 文件发生过临时改动，且已正确复原。
   - 确认无测试照片残留。
5. **进入 Codex 只读失败审查阶段**，分析数据差异的原因，待下一步规划中统一处理。
